# 架构概述

CoralNPU 是一个专为机器学习推理设计的硬件加速器，基于 32 位 RISC-V 指令集架构（ISA）。它是由 Google Research 设计的开源 IP，专门针对超低功耗的可穿戴设备（如耳戴式设备、AR 眼镜和智能手表）进行优化。

## 核心设计理念

CoralNPU 的设计从领域特定的矩阵计算能力开始，然后添加向量（SIMD）和标量处理能力，形成一个融合设计。这种设计理念使得 CoralNPU 能够高效地执行机器学习推理任务。

**解释：什么是融合设计？**

融合设计是指将多种处理单元（标量、向量、矩阵）集成在一个处理器中，它们共享资源并协同工作。这样可以避免数据在不同处理器之间传输的开销，提高效率。

## 三大处理组件

CoralNPU 包含三个不同的处理器组件，它们协同工作：



- **标量处理器（Scalar Core）**
  

  - 基于 RISC-V rv32imf 指令集
  - 负责控制流、地址计算和循环管理
  - 驱动向量和矩阵单元的命令队列
  


- **向量处理器（Vector Core / SIMD）**
  

  - 128 位 SIMD 流水线（未来支持 256 位）
  - 64 个向量寄存器（v0-v63），每个 256 位宽
  - 支持 8、16、32 位数据宽度
  - 处理向量运算和数据搬运
  


- **矩阵处理器（MAC Engine）**
  

  - 量化外积乘累加引擎
  - 每周期执行 256 次乘累加操作（MAC）
  - 8x8 累加器阵列，每个累加器 32 位
  




**解释：什么是 SIMD？**

SIMD（Single Instruction Multiple Data，单指令多数据）是一种并行计算技术。一条指令可以同时对多个数据进行相同的操作。比如，一条向量加法指令可以同时对 8 个数进行加法，而不是执行 8 次单独的加法指令。

## 关键特性



- **指令集**：RV32IMF\_Zve32x（具体为 rv32imf\_zve32x\_zicsr\_zifencei\_zbb）
- **地址空间**：32 位地址空间
- **流水线**：四级流水线，顺序派发，乱序退休
- **派发能力**：每周期最多派发 4 条标量指令，2 条向量指令
- **紧耦合内存**：
  

  - 8 KB ITCM（指令紧耦合内存）
  - 32 KB DTCM（数据紧耦合内存）
  - 单周期访问延迟，比缓存更高效
  

- **总线接口**：AXI4 总线接口，可作为主设备和从设备



**解释：什么是紧耦合内存（TCM）？**

紧耦合内存是直接连接到处理器核心的小容量、高速内存。它的访问延迟是固定的（通常是 1 个时钟周期），不像缓存那样有命中/未命中的不确定性。这对于实时性要求高的应用非常重要。

**解释：什么是顺序派发、乱序退休？**



- 顺序派发：指令按照程序顺序被发送到执行单元
- 乱序退休：指令可以按照完成的顺序（而不是程序顺序）写回结果



这样可以提高处理器的效率，因为不需要等待前面的慢指令完成。

**解释：什么是 ALU、BRU、MLU、DVU、LSU、CSR？**

这些是标量核心中的执行单元：


- ALU (Arithmetic Logic Unit)：算术逻辑单元，执行加减、逻辑运算等
- BRU (Branch Unit)：分支单元，处理跳转指令
- MLU (Multiply Unit)：乘法单元
- DVU (Divide Unit)：除法单元
- LSU (Load/Store Unit)：加载/存储单元，处理内存访问
- CSR (Control and Status Register)：控制和状态寄存器单元



## 流水线结构

CoralNPU 采用四级流水线设计，下面是流水线各阶段的详细说明：

```

│                        CoralNPU 流水线结构                           │
├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  阶段 1: 取指 (Instruction Fetch)                                    │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  • 从 ITCM 或 I-Cache 取指令                                 │   │
```

**指令延迟表**

不同类型的指令在执行阶段需要不同的时钟周期：



**指令类型** & **延迟（周期）** & **说明** \\

ALU & 1 & 加、减、异或等 \\

CSR & 1 & CSR 指令 \\

BRU & 1 & 分支、跳转、断点等 \\

MLU & 2 & 乘法指令 \\

DVU & 可变 & 除法、取余指令 \\

LSU & 2+ & 加载、存储指令 \\



**解释：什么是记分牌（Scoreboard）？**

记分牌是一种硬件机制，用于跟踪哪些寄存器正在被使用（即有指令正在写入该寄存器）。当一条指令需要读取某个寄存器时，记分牌会检查该寄存器是否正在被写入。如果是，该指令就需要等待，避免读到旧的数据。这是解决数据冒险（data hazard）的一种方法。

**解释：什么是互锁（Interlock）？**

互锁是指当检测到数据依赖时，流水线会暂停（stall）后续指令的派发，直到依赖被解决。比如，如果指令 B 需要使用指令 A 的结果，但指令 A 还没有完成，那么指令 B 就会被互锁，等待指令 A 完成。

## 典型机器学习推理数据流

下面是一个典型的卷积神经网络推理任务的数据流示例：

```
步骤 1: 数据加载
  外部内存 → AXI4 → D-Cache → DTCM → 向量寄存器
  (加载输入特征图和卷积核权重)

步骤 2: 标量核心控制
  标量核心执行循环控制代码
  计算地址偏移
  向向量核心发送向量指令

步骤 3: 向量数据准备
  向量核心从 DTCM 加载数据到向量寄存器
  数据重排和对齐

步骤 4: 矩阵乘累加
  向量寄存器 → MAC 引擎
  执行外积运算（卷积计算）
  结果累加到累加器阵列

步骤 5: 后处理
  累加器结果 → 向量寄存器
  向量核心执行激活函数（如 ReLU）
  量化和饱和处理

步骤 6: 结果存储
  向量寄存器 → DTCM → D-Cache → AXI4 → 外部内存
  (存储输出特征图)
```

**解释：什么是外积运算？**

外积（Outer Product）是一种矩阵运算。对于两个向量 A (大小为 m) 和 B (大小为 n)，它们的外积是一个 m×n 的矩阵。在 CoralNPU 中，MAC 引擎使用外积结构来最大化计算效率：一个向量在一个维度上广播（卷积权重），另一个向量在另一个维度上广播（输入数据），这样可以用最少的内存访问完成最多的计算。

## 标量和向量的解耦设计

CoralNPU 采用了解耦设计，标量前端和向量后端通过 FIFO 缓冲区连接：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      标量-向量解耦机制                               │
├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  标量核心 (快速派发)                                                  │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  • 每周期可派发 4 条标量指令                                 │   │
│  │  • 向量指令被快速派发到 FIFO                                 │   │
│  │  • 不等待向量指令完成                                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ↓                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │              向量指令 FIFO (缓冲队列)                        │   │
│  │  • 缓冲标量派发的向量指令                                    │   │
│  │  • 检查向量寄存器依赖关系                                    │   │
│  │  • 当依赖解决后才发送到向量执行单元                          │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ↓                                         │
│  向量核心 (独立执行)                                                  │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  • 从 FIFO 取出就绪的向量指令                                │   │
│  │  • 每周期可执行 2 条向量指令                                 │   │
│  │  • 独立于标量核心的执行进度                                  │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**解释：为什么要解耦？**

解耦设计的好处是：


- 标量核心可以快速派发指令，不会被慢速的向量运算阻塞
- 向量核心可以独立执行，充分利用硬件资源
- FIFO 缓冲区可以隐藏延迟，提高整体吞吐量
- 标量核心可以提前执行循环控制和地址计算，为向量运算准备好数据



这就像工厂的流水线：前端快速下订单（标量派发），中间有个订单池（FIFO），后端按照订单独立生产（向量执行）。

```
│                                                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  累加器阵列                                                  │   │
│  │  • 8x8 个累加器                                              │   │
│  │  • 每个 32 位                                                │   │
│  │  • 访问延迟: 0 周期                                          │   │
│  │  • 总容量: 2 KB                                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ↓                                         │
│  层次 1: 紧耦合内存 TCM (快速，容量小)                                │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  ITCM (指令紧耦合内存)                                       │   │
│  │  • 容量: 8 KB                                                │   │
│  │  • 访问延迟: 1 周期 (固定)                                   │   │
│  │  • 用途: 存储程序代码                                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  DTCM (数据紧耦合内存)                                       │   │
│  │  • 容量: 32 KB                                               │   │
│  │  • 访问延迟: 1 周期 (固定)                                   │   │
│  │  • 用途: 存储常用数据、权重、中间结果                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ↓                                         │
│  层次 2: L1 缓存 (中速，容量中等)                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  L1 I-Cache (指令缓存)                                       │   │
│  │  • 容量: 8 KB                                                │   │
│  │  • 组相联: 4-way                                             │   │
│  │  • 缓存行: 256 位                                            │   │
│  │  • 访问延迟: 1-2 周期 (命中时)                               │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  L1 D-Cache (数据缓存)                                       │   │
│  │  • 容量: 16 KB (双 bank，每个 8 KB)                          │   │
│  │  • 组相联: 4-way                                             │   │
│  │  • 缓存行: 256 位                                            │   │
│  │  • 访问延迟: 1-2 周期 (命中时)                               │   │
│  │  • 支持下一行预取                                            │   │
│  │  • 支持缓存行刷新                                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ↓                                         │
│  层次 3: 外部内存 (慢速，容量大)                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  外部 DRAM                                                   │   │
│  │  • 容量: 通常 MB-GB 级别                                     │   │
│  │  • 访问延迟: 数十到数百周期                                  │   │
│  │  • 通过 AXI4 总线访问                                        │   │
│  │  • 用途: 存储模型、大数据集、输入输出                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

## 存储层次特性对比



**存储层次** & **容量** & **访问延迟** & **带宽** & **确定性** & **主要用途** \\

标量寄存器 & 124 B & 0 周期 & 最高 & 确定 & 临时变量、循环计数器 \\

向量寄存器 & 2 KB & 0 周期 & 最高 & 确定 & 向量数据、中间结果 \\

累加器 & 2 KB & 0 周期 & 最高 & 确定 & 矩阵乘累加结果 \\

ITCM & 8 KB & 1 周期 & 很高 & 确定 & 热点代码、循环体 \\

DTCM & 32 KB & 1 周期 & 很高 & 确定 & 权重、激活值、中间数据 \\

L1 I-Cache & 8 KB & 1-2 周期 & 高 & 不确定 & 指令缓存 \\

L1 D-Cache & 16 KB & 1-2 周期 & 高 & 不确定 & 数据缓存、对齐缓冲 \\

外部 DRAM & MB-GB & 数十-数百周期 & 低 & 不确定 & 模型、数据集 \\



**解释：什么是确定性？**

确定性是指访问延迟是否固定。TCM 的访问延迟总是 1 个周期，这是确定的；而 Cache 可能命中（快）也可能未命中（慢），延迟不确定。对于实时系统，确定性很重要，因为可以精确预测程序的执行时间。


## 存储优化策略

为了最大化性能，CoralNPU 采用了以下存储优化策略：



- **数据局部性优化**
  

  - 将频繁访问的数据放在 DTCM 中
  - 利用向量寄存器保存中间结果，减少内存访问
  - 分块处理大数据，提高缓存命中率
  


- **双 Bank D-Cache**
  

  - L1 D-Cache 分为两个 8KB 的 bank
  - 支持下一行预取，隐藏内存延迟
  - 为标量和向量访问提供更高的带宽
  


- **流式数据访问**
  

  - 对于大模型，权重和数据流式加载
  - 避免反复访问外部内存
  - 利用 DMA 进行后台数据传输
  


- **缓存刷新控制**
  

  - 支持缓存行刷新和全刷新
  - 确保数据一致性
  - 核心在刷新完成前会暂停，简化编程模型
  




**解释：什么是数据局部性？**

数据局部性是指程序倾向于访问最近访问过的数据（时间局部性）或相邻的数据（空间局部性）。利用数据局部性可以提高缓存命中率，减少访问慢速内存的次数。例如，在处理图像时，相邻的像素通常会被连续访问，这就是空间局部性。

**解释：什么是 DMA？**

DMA（Direct Memory Access，直接内存访问）是一种硬件机制，允许外设直接访问内存，而不需要 CPU 参与每次数据传输。这样 CPU 可以继续执行其他任务，而 DMA 控制器在后台搬运数据，提高了系统的整体效率。

# 关键技术特性

## 条带挖掘（Stripmining）

条带挖掘是 CoralNPU 的一个重要特性，用于减少指令派发压力。

```
┌─────────────────────────────────────────────────────────────────────┐
│                        条带挖掘机制                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  标量前端派发 1 条向量指令:                                           │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  vadd v0    (单条指令派发)                                   │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ↓                                         │
│  向量后端展开为 4 条指令:                                             │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  vadd v0    (第 1 个向量寄存器)                              │   │
│  │  vadd v1    (第 2 个向量寄存器)                              │   │
│  │  vadd v2    (第 3 个向量寄存器)                              │   │
│  │  vadd v3    (第 4 个向量寄存器)                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                            ↓                                         │
│  向量执行单元依次执行:                                                │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  周期 1: 执行 vadd v0                                        │   │
│  │  周期 2: 执行 vadd v1                                        │   │
│  │  周期 3: 执行 vadd v2                                        │   │
│  │  周期 4: 执行 vadd v3                                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**解释：为什么需要条带挖掘？**

条带挖掘（Stripmining）的定义是：将基于数组的并行性折叠以适应可用的硬件并行性。

在 CoralNPU 中，向量寄存器有 64 个（v0-v63），但如果每次操作都需要派发 4 条指令，会给前端带来很大压力。条带挖掘允许前端只派发 1 条指令，后端自动展开为 4 条，这样：


- 减少了前端的派发压力
- 原生支持指令级的分块模式
- 提高了指令吞吐量



这就像批量处理：你只需要下一个订单"加工 4 个零件"，工厂会自动处理这 4 个零件，而不需要你下 4 次订单。

## 分支预测策略

CoralNPU 采用简单但有效的静态分支预测策略：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        分支预测策略                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  向后跳转 (Backward Branch):                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  预测: taken (跳转)                                          │   │
│  │  原因: 通常是循环，大概率会跳转                              │   │
│  │                                                              │   │
│  │  示例:                                                       │   │
│  │    loop:                                                     │   │
│  │      addi x1, x1, 1                                          │   │
│  │      blt x1, x10, loop  ← 向后跳转，预测 taken              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  向前跳转 (Forward Branch):                                          │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  预测: not-taken (不跳转)                                    │   │
│  │  原因: 通常是条件判断，大概率顺序执行                        │   │
│  │                                                              │   │
│  │  示例:                                                       │   │
│  │    beq x1, x2, skip  ← 向前跳转，预测 not-taken             │   │
│  │    addi x3, x3, 1                                            │   │
│  │    skip:                                                     │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
│  预测错误的代价:                                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  • 流水线需要刷新                                            │   │
│  │  • 损失 1 个周期                                             │   │
│  │  • 重新从正确的地址取指                                      │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**解释：为什么这样预测？**

这种预测策略基于程序的统计特性：


- 向后跳转通常是循环，循环体会执行多次，所以大概率会跳转
- 向前跳转通常是 if 语句，大多数情况下条件不满足，顺序执行



虽然简单，但对于机器学习推理这种循环密集的任务，这种策略的准确率很高。

## 外积 MAC 引擎

CoralNPU 的核心是外积乘累加引擎，专为卷积运算优化：

```
┌─────────────────────────────────────────────────────────────────────┐
│                      外积 MAC 引擎结构                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
│  输入 A (权重向量，"宽"维度):                                         │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  a0  a1  a2  a3  a4  a5  a6  a7                              │   │
│  └─────────────────────────────────────────────────────────────┘   │
│         │   │   │   │   │   │   │   │                              │
│         ↓   ↓   ↓   ↓   ↓   ↓   ↓   ↓  (广播到每一行)              │
│                                                                       │
│  输入 B (激活值，"窄"维度):                                           │
│  ┌───┐                                                               │
│  │b0 │→  [a0*b0] [a1*b0] [a2*b0] [a3*b0] [a4*b0] [a5*b0] [a6*b0] [a7*b0] │
│  ├───┤                                                               │
│  │b1 │→  [a0*b1] [a1*b1] [a2*b1] [a3*b1] [a4*b1] [a5*b1] [a6*b1] [a7*b1] │
│  ├───┤                                                               │
│  │b2 │→  [a0*b2] [a1*b2] [a2*b2] [a3*b2] [a4*b2] [a5*b2] [a6*b2] [a7*b2] │
│  ├───┤                                                               │
│  │b3 │→  [a0*b3] [a1*b3] [a2*b3] [a3*b3] [a4*b3] [a5*b3] [a6*b3] [a7*b3] │
│  ├───┤                                                               │
│  │b4 │→  [a0*b4] [a1*b4] [a2*b4] [a3*b4] [a4*b4] [a5*b4] [a6*b4] [a7*b4] │
│  ├───┤                                                               │
│  │b5 │→  [a0*b5] [a1*b5] [a2*b5] [a3*b5] [a4*b5] [a5*b5] [a6*b5] [a7*b5] │
│  ├───┤                                                               │
│  │b6 │→  [a0*b6] [a1*b6] [a2*b6] [a3*b6] [a4*b6] [a5*b6] [a6*b6] [a7*b6] │
│  ├───┤                                                               │
│  │b7 │→  [a0*b7] [a1*b7] [a2*b7] [a3*b7] [a4*b7] [a5*b7] [a6*b7] [a7*b7] │
│  └───┘                                                               │
│   ↑                                                                  │
│  (广播到每一列)                                                       │
│                                                                       │
│  结果: 8x8 累加器阵列                                                 │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  每个累加器累加多次 MAC 结果                                 │   │
│  │  每周期执行: 8 x 8 x 4 = 256 次 MAC 操作                    │   │
│  │  (4 来自每个 8 位乘法单元处理 4 个 8 位数据)                 │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

**解释：为什么使用外积结构？**

外积结构的优势：


- **最大化计算效率**：一次读取 8 个权重和 8 个激活值，可以执行 64 次乘法
- **减少内存访问**：通过广播机制，每个数据被多次使用，减少了内存带宽需求
- **适合卷积**：卷积运算天然就是外积形式，权重在一个维度上重复使用，输入在另一个维度上重复使用



这就像矩阵乘法：不需要反复读取相同的数据，而是一次读取后在多个计算中重复使用。

# 总结

CoralNPU 是一个专为机器学习推理优化的硬件加速器，其架构设计体现了以下核心思想：



- **融合设计**：将标量、向量、矩阵三种处理能力融合在一个处理器中，避免数据传输开销。

- **解耦流水线**：标量前端和向量后端通过 FIFO 解耦，提高指令吞吐量和执行效率。

- **多层次存储**：从寄存器到 TCM 到 Cache 到外部内存，提供了灵活的存储层次，平衡了速度和容量。

- **外积 MAC 引擎**：专为卷积运算设计的外积结构，最大化了计算效率和内存带宽利用率。

- **低功耗优化**：紧耦合内存、简单的分支预测、顺序派发等设计都是为了降低功耗，适合可穿戴设备。

- **开源和可扩展**：作为开源 IP，CoralNPU 可以灵活集成到各种 SoC 中，满足不同的应用需求。



通过这些设计，CoralNPU 能够在超低功耗的约束下，提供高效的机器学习推理性能，非常适合耳戴式设备、AR 眼镜、智能手表等可穿戴设备。
