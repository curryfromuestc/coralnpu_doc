# 存储系统

## 存储系统概述

CoralNPU 的存储系统采用分层设计，旨在为机器学习加速器提供高效的数据访问。存储层次从快到慢依次为：

```
                    +------------------+
                    |   CPU Core       |
                    +------------------+
                           |
        +------------------+------------------+
        |                                     |
   +----------+                          +----------+
   |   ITCM   |                          |   DTCM   |
   |   8KB    |                          |   32KB   |
   +----------+                          +----------+
        |                                     |
        |                                     |
   +----------+                          +----------+
   | L1 I$    |                          | L1 D$    |
   | 8KB 4路  |                          | 16KB 4路 |
   +----------+                          +----------+
        |                                     |
        +------------------+------------------+
                           |
                    +------------------+
                    |   AXI 总线       |
                    +------------------+
                           |
                    +------------------+
                    |   外部存储器     |
                    +------------------+
```

CoralNPU 的存储系统有以下特点：



- **双路径设计**：指令和数据分别有独立的存储路径（哈佛架构）
- **TCM + Cache 混合架构**：同时使用紧耦合内存和缓存
- **低延迟优先**：TCM 提供单周期访问，适合实时性要求高的场景
- **灵活性**：Cache 提供更大的地址空间和自动数据管理



## 紧耦合内存 (TCM)

### 什么是 TCM？

TCM (Tightly Coupled Memory，紧耦合内存) 是一种直接连接到 CPU 核心的高速存储器。与普通的缓存不同，TCM 有以下特点：



- **确定性访问延迟**：访问 TCM 总是固定的延迟（通常是单周期），不会出现缓存未命中
- **直接映射**：TCM 有固定的地址范围，软件可以直接控制哪些数据放在 TCM 中
- **无需缓存管理**：不需要缓存一致性协议，不需要刷新操作
- **低功耗**：没有标签匹配逻辑，功耗比缓存低



### 为什么使用 TCM？

在机器学习加速器中，TCM 特别有用：



- **实时性保证**：神经网络推理需要可预测的执行时间，TCM 的确定性延迟很重要
- **小代码段**：ML 推理的控制代码通常很小，8KB ITCM 足够存放核心循环
- **临时数据**：DTCM 可以存放循环计数器、地址指针等频繁访问的变量
- **降低功耗**：对于频繁访问的数据，TCM 比 Cache 更省电



### TCM vs Cache 对比



**特性** & **TCM** & **Cache** \\

访问延迟 & 固定（1 周期） & 可变（命中 1-2 周期，未命中几十周期） \\

地址空间 & 固定小范围 & 整个地址空间 \\

软件控制 & 完全可控 & 硬件自动管理 \\

功耗 & 低 & 较高 \\

面积 & 小 & 较大（需要标签阵列） \\

适用场景 & 关键代码和数据 & 通用访问 \\



**为什么 CoralNPU 同时使用两者？**

CoralNPU 采用 TCM + Cache 混合架构的原因：



- TCM 用于**关键路径**：存放推理循环、中断处理等对延迟敏感的代码和数据
- Cache 用于**灵活性**：处理大数据集、动态加载的模型权重等
- 软件可以根据需求选择：将关键数据放在 TCM，其他数据使用 Cache



## 指令 TCM (ITCM)

ITCM 用于存储指令代码，直接连接到取指单元。

### 基本参数



- **容量**：8KB (8192 字节)
- **地址范围**：0x0000\_0000 - 0x0000\_1FFF
- **访问延迟**：单周期
- **数据宽度**：256 位（每次可取 8 条 32 位指令）
- **实现方式**：SRAM



### 工作原理

```
取指单元 (Fetch Unit)
    |
    | 32-bit 地址
    v
+-------------------+
|   地址译码        |
+-------------------+
    |
    | 选择 SRAM 行
    v
+-------------------+
|   ITCM SRAM      |
|   (256 行)       |
+-------------------+
    |
    | 256-bit 数据
    v
+-------------------+
|   指令缓冲       |
+-------------------+
```

ITCM 的访问流程：



- 取指单元发出 32 位地址
- 地址的高位用于选择 SRAM 的行（256 行，需要 8 位地址）
- 一次读取返回 256 位数据（8 条指令）
- 指令缓冲区缓存这些指令供译码使用



### 使用建议

ITCM 适合存放：



- **启动代码**：系统初始化代码
- **中断处理程序**：需要快速响应的中断
- **推理循环**：神经网络推理的核心循环
- **频繁调用的函数**：如矩阵乘法、激活函数等



示例（链接脚本）：

```
MEMORY
{
    ITCM (rx)  : ORIGIN = 0x00000000, LENGTH = 8K
    DTCM (rw)  : ORIGIN = 0x00010000, LENGTH = 32K
}

SECTIONS
{
    .text : {
        *(.text.startup)    /* 启动代码 */
        *(.text.critical)   /* 关键函数 */
        *(.text)            /* 其他代码 */
    } > ITCM
}
```

## 数据 TCM (DTCM)

DTCM 用于存储数据，直接连接到加载存储单元 (LSU)。

### 基本参数



- **容量**：32KB (32768 字节)
- **地址范围**：0x0001\_0000 - 0x0001\_7FFF
- **访问延迟**：单周期
- **数据宽度**：128 位（支持向量访问）
- **实现方式**：SRAM，支持字节写掩码



### 工作原理

```
加载存储单元 (LSU)
    |
    | 地址 + 数据 + 写掩码
    v
+-------------------+
|   地址译码        |
+-------------------+
    |
    v
+-------------------+
|   DTCM SRAM      |
|   (2048 行)      |
|   每行 128 位    |
+-------------------+
    |
    | 读数据 / 写数据
    v
+-------------------+
|   数据对齐       |
+-------------------+
```

DTCM 的特殊功能：



- **字节写掩码**：支持字节、半字、字的部分写入
- **向量访问**：支持 128 位宽的向量数据访问
- **无对齐限制**：硬件自动处理非对齐访问（可能需要多周期）



### 使用建议

DTCM 适合存放：



- **栈空间**：函数调用栈
- **全局变量**：频繁访问的全局变量
- **临时缓冲区**：循环中的临时数组
- **向量寄存器溢出**：当向量寄存器不够用时的溢出空间



示例（C 代码）：

```
// 将变量放在 DTCM 中
__attribute__((section(".dtcm")))
int32_t activation_buffer[1024];

// 将栈放在 DTCM 中
__attribute__((section(".stack")))
uint8_t stack_space[4096];
```

## L1 指令缓存 (L1 I-Cache)

当程序访问 ITCM 范围之外的指令时，会使用 L1 指令缓存。

### 基本参数



- **容量**：8KB
- **组织方式**：4 路组相联
- **缓存行大小**：32 字节（256 位）
- **总行数**：256 行（每路 64 行）
- **替换策略**：伪 LRU (Least Recently Used)



### 缓存结构

```
地址格式 (32 位):
+--------+-------+--------+
|  Tag   | Index | Offset |
| [31:11]| [10:5]| [4:0]  |
+--------+-------+--------+
   21位    6位     5位

缓存组织:
Set 0:  [Way 0] [Way 1] [Way 2] [Way 3]
Set 1:  [Way 0] [Way 1] [Way 2] [Way 3]
...
Set 63: [Way 0] [Way 1] [Way 2] [Way 3]

每个 Way 包含:
- Valid 位: 1 位
- Tag: 21 位
- Data: 256 位 (32 字节)
```

### 工作原理

**缓存命中流程**：



- 取指单元发出地址
- 使用 Index (地址的 [10:5] 位) 选择一个 Set
- 并行比较 4 个 Way 的 Tag
- 如果某个 Way 的 Tag 匹配且 Valid=1，缓存命中
- 返回对应的 256 位数据



**缓存未命中流程**：



- 如果所有 Way 都不匹配，缓存未命中
- 根据 LRU 策略选择一个 Way 进行替换
- 通过 AXI 总线从外部存储器读取数据
- 将新数据写入选中的 Way
- 更新 Valid 位和 Tag
- 返回数据给取指单元



### 替换策略

L1 I-Cache 使用**伪 LRU** 替换策略：



- 每个 Set 维护一个访问历史计数器
- 每次访问时，被访问的 Way 计数器设为最大值
- 其他 Way 的计数器递减
- 替换时选择计数器值最小的 Way



这种策略比真正的 LRU 简单，但效果接近。

### 性能特点



- **命中延迟**：1 周期（与 ITCM 相同）
- **未命中惩罚**：取决于外部存储器延迟（通常 10-50 周期）
- **命中率**：对于循环代码，命中率通常 > 95\%




## L1 数据缓存 (L1 D-Cache)

L1 数据缓存用于缓存 DTCM 范围之外的数据访问。

### 基本参数



- **容量**：16KB（每个 Bank 8KB）
- **组织方式**：4 路组相联，双 Bank 结构
- **缓存行大小**：32 字节（256 位）
- **总行数**：512 行（每 Bank 256 行）
- **替换策略**：伪 LRU
- **写策略**：写回 (Write-back) + 写分配 (Write-allocate)



### 双 Bank 结构

L1 D-Cache 的独特设计是**双 Bank 架构**：

```
                LSU
                 |
        +--------+--------+
        |                 |
    +-------+         +-------+
    | Bank 0|         | Bank 1|
    | 8KB   |         | 8KB   |
    | 4-way |         | 4-way |
    +-------+         +-------+
        |                 |
        +--------+--------+
                 |
            AXI 总线
```

**Bank 选择**：



- 地址的某一位（通常是 bit[5]）用于选择 Bank
- Bank 0: 地址 bit[5] = 0
- Bank 1: 地址 bit[5] = 1



**双 Bank 的优势**：



- **跨行访问**：当一次访问跨越两个缓存行时，可以同时访问两个 Bank
- **预取**：可以在访问 Bank 0 的同时预取 Bank 1 的下一行
- **带宽加倍**：理论上可以提供 2 倍的访问带宽



### 地址映射

```
地址格式 (32 位):
+--------+-------+------+--------+
|  Tag   | Index |Bank  | Offset |
| [31:12]| [11:6]| [5]  | [4:0]  |
+--------+-------+------+--------+
   20位     6位    1位     5位

注意：Bank 位从地址中移除后再进行 Index 计算
实际 Index = {addr[11:6]}
```

### 写策略

L1 D-Cache 使用**写回 + 写分配**策略：

**写回 (Write-back)**：


- 写操作只更新缓存，不立即写入外部存储器
- 每个缓存行有一个 Dirty 位，标记是否被修改
- 当 Dirty 行被替换时，才写回外部存储器



**写分配 (Write-allocate)**：


- 写未命中时，先将数据从外部存储器读入缓存
- 然后在缓存中执行写操作



**优点**：


- 减少外部存储器访问次数
- 提高写操作性能
- 适合频繁读写的数据



### 缓存一致性

L1 D-Cache 支持软件控制的缓存刷新：

```
// 刷新单个缓存行
void flush_dcache_line(void *addr);

// 刷新整个缓存
void flush_dcache_all(void);

// 使缓存行无效
void invalidate_dcache_line(void *addr);
```

**使用场景**：


- DMA 传输前后需要刷新缓存
- 与外部设备共享内存时
- 多核系统中的数据同步



### 性能优化

**预取机制**：


- 当访问地址 X 时，如果 X+32 在另一个 Bank，自动预取
- 预取条件：地址在 4KB 页内（避免跨页预取）



**写合并**：


- 多个写操作可以合并到同一缓存行
- 减少外部存储器写次数



## SRAM 设计

CoralNPU 的 TCM 和 Cache 都使用 SRAM 实现。

### SRAM 类型

CoralNPU 使用两种 SRAM：



- **Sram\_1rw\_256x256**：用于 L1 I-Cache
   

   - 256 行，每行 256 位
   - 单端口读写
   


- **Sram\_1rwm\_256x288**：用于 L1 D-Cache
   

   - 256 行，每行 288 位（256 位数据 + 32 位掩码）
   - 支持字节写掩码
   


- **Sram\_Nx128**：用于 TCM
   

   - 可配置行数
   - 每行 128 位
   - 支持字节写掩码
   




### SRAM 接口

```
class Sram_1rw_256x256 extends BlackBox {
  val io = IO(new Bundle {
    val clock    = Input(Clock())      // 时钟
    val valid    = Input(Bool())       // 使能信号
    val write    = Input(Bool())       // 写使能
    val addr     = Input(UInt(8.W))    // 地址 (256 行)
    val wdata    = Input(UInt(256.W))  // 写数据
    val rdata    = Output(UInt(256.W)) // 读数据
    val volt_sel = Input(Bool())       // 电压选择
  })
}
```

### 电压选择

SRAM 支持**双电压模式**：



- **高性能模式**：volt\_sel = 0，更高的电压，更快的访问速度
- **低功耗模式**：volt\_sel = 1，更低的电压，降低功耗



这个功能用于动态电压频率调节 (DVFS)。

### 时序特性

```
时钟周期:
     ___     ___     ___
CLK |   |___|   |___|   |___

读操作:
     _______________
valid               |_______
     _______________
addr  ==== A0 ======|=======
                    _______
rdata =============>| D0   |
     <-- 1 cycle -->

写操作:
     _______________
valid               |_______
     _______________
write               |_______
     _______________
addr  ==== A0 ======|=======
     _______________
wdata ==== D0 ======|=======
```

SRAM 是**同步 SRAM**：


- 读操作：地址在当前周期给出，数据在下一周期返回
- 写操作：地址和数据在同一周期给出，下一周期完成写入



### 字节写掩码

L1 D-Cache 的 SRAM 支持字节写掩码：

```
288 位 SRAM = 256 位数据 + 32 位掩码

数据位: [287:32] - 256 位数据
掩码位: [31:0]   - 32 位掩码 (每位对应 8 位数据)

写掩码示例:
wmask = 0x0000000F  // 只写入最低 4 字节
wmask = 0xFFFFFFFF  // 写入全部 32 字节
```

这个功能用于支持字节、半字、字的部分写入。

### 地址空间特性



**地址范围** & **名称** & **大小** & **缓存** & **用途** \\

0x0000\_0000 - 0x0000\_1FFF & ITCM & 8KB & 否 & 关键指令代码 \\

0x0001\_0000 - 0x0001\_7FFF & DTCM & 32KB & 否 & 关键数据 \\

0x0003\_0000 - 0x0003\_0FFF & CSR & 4KB & 否 & 控制状态寄存器 \\

0x1000\_0000 - 0x1000\_7FFF & ROM & 32KB & 是 & 启动代码 \\

0x2000\_0000 - 0x203F\_FFFF & SRAM & 4MB & 是 & 通用数据 \\

0x4000\_0000 - 0x4FFF\_FFFF & 外设 & 256MB & 否 & 外设寄存器 \\

0x8000\_0000 - 0xFFFF\_FFFF & DDR & 2GB & 是 & 大容量存储 \\



### 访问路由

CPU 根据地址自动选择访问路径：

```
地址判断流程:

if (addr < 0x0000_2000)
    -> 访问 ITCM (单周期)
else if (addr >= 0x0001_0000 && addr < 0x0001_8000)
    -> 访问 DTCM (单周期)
else if (addr >= 0x0003_0000 && addr < 0x0003_1000)
    -> 访问 CSR (单周期)
else if (addr >= 0x4000_0000 && addr < 0x5000_0000)
    -> 访问外设 (通过 AXI，多周期)
else
    -> 访问 Cache (命中 1-2 周期，未命中多周期)
       -> Cache 未命中时访问外部存储器 (通过 AXI)
```

### 编程注意事项

**1. 链接脚本配置**

```
MEMORY
{
    ITCM (rx)  : ORIGIN = 0x00000000, LENGTH = 8K
    DTCM (rw)  : ORIGIN = 0x00010000, LENGTH = 32K
    SRAM (rwx) : ORIGIN = 0x20000000, LENGTH = 4M
}

SECTIONS
{
    .text : { *(.text*) } > ITCM
    .data : { *(.data*) } > DTCM
    .bss  : { *(.bss*)  } > DTCM
    .heap : { . = ALIGN(8); _heap_start = .; } > SRAM
}
```

**2. 数据放置**

```
// 放在 ITCM 中的函数
__attribute__((section(".itcm")))
void critical_function(void) {
    // 关键代码
}

// 放在 DTCM 中的变量
__attribute__((section(".dtcm")))
int32_t fast_buffer[1024];

// 放在 SRAM 中的大数组
__attribute__((section(".sram")))
uint8_t large_buffer[1024*1024];
```

**3. 缓存管理**

```
// 在 DMA 传输前刷新缓存
flush_dcache_range(buffer, size);

// DMA 传输
dma_transfer(buffer, size);

// DMA 完成后使缓存无效
invalidate_dcache_range(buffer, size);
```

### 性能考虑

**访问延迟对比**：



**存储类型** & **命中延迟** & **未命中惩罚** & **带宽** \\

ITCM & 1 周期 & N/A & 256 位/周期 \\

DTCM & 1 周期 & N/A & 128 位/周期 \\

L1 I\$ & 1 周期 & 10-50 周期 & 256 位/周期 \\

L1 D\$ & 1-2 周期 & 10-50 周期 & 256 位/周期 \\

SRAM & 10-20 周期 & N/A & 128 位/周期 \\

DDR & 50-100 周期 & N/A & 128 位/周期 \\



**优化建议**：



- **热点代码放 ITCM**：将最频繁执行的循环放在 ITCM 中
- **热点数据放 DTCM**：将循环变量、累加器等放在 DTCM 中
- **顺序访问**：利用缓存行预取，顺序访问比随机访问快
- **对齐访问**：对齐的访问比非对齐访问快
- **批量传输**：使用 DMA 进行大块数据传输
