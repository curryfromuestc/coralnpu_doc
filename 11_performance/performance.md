# 第十一章：性能

## 11.1 性能概述

### 11.1.1 什么是处理器性能

**处理器性能**是指处理器完成任务的速度和效率。对于初学者来说，可以从以下几个角度理解：

**基本概念**：
```
性能 = 工作量 / 时间

例如：
- 一个处理器 1 秒钟能执行 1000 条指令
- 另一个处理器 1 秒钟能执行 2000 条指令
- 那么第二个处理器的性能是第一个的 2 倍
```

**类比理解**：
- **处理器**就像是一个工厂的生产线
- **指令**就像是需要加工的零件
- **性能**就是单位时间内能加工多少零件

### 11.1.2 性能指标

处理器性能有多个衡量指标，每个指标关注不同的方面：

#### IPC（Instructions Per Cycle，每周期指令数）

**定义**：平均每个时钟周期能执行多少条指令。

```
IPC = 执行的指令总数 / 消耗的时钟周期数

例如：
- 执行了 1000 条指令
- 用了 500 个时钟周期
- IPC = 1000 / 500 = 2.0
```

**IPC 的意义**：
- IPC = 1.0：每个周期执行 1 条指令（单发射）
- IPC = 2.0：每个周期平均执行 2 条指令（双发射）
- IPC = 4.0：每个周期平均执行 4 条指令（四发射）

**影响 IPC 的因素**：
```
1. 指令级并行性（ILP）
   - 程序中有多少指令可以同时执行
   - 例如：a = b + c; d = e + f; 这两条可以并行

2. 数据依赖
   - 后面的指令需要前面指令的结果
   - 例如：a = b + c; d = a + e; 第二条必须等第一条完成

3. 控制依赖
   - 分支指令导致的流水线停顿
   - 例如：if (a > b) { ... } 需要等待条件判断

4. 资源冲突
   - 多条指令竞争同一个硬件单元
   - 例如：两条乘法指令但只有一个乘法器
```

#### 吞吐量（Throughput）

**定义**：单位时间内能处理的数据量或任务数。

```
吞吐量 = 完成的任务数 / 时间

例如：
- 1 秒钟处理了 100 张图片
- 吞吐量 = 100 图片/秒
```

**吞吐量的应用场景**：
- **批处理任务**：图像处理、视频编码
- **机器学习推理**：每秒能推理多少个样本
- **数据处理**：每秒能处理多少个数据包

#### 延迟（Latency）

**定义**：完成一个任务所需的时间。

```
延迟 = 任务完成时间 - 任务开始时间

例如：
- 一条加法指令从开始到完成需要 1 个周期
- 一条乘法指令从开始到完成需要 2 个周期
- 一条除法指令从开始到完成需要 10 个周期
```

**延迟的类型**：
```
1. 指令延迟
   - 单条指令的执行时间
   - ALU 指令：1 周期
   - 乘法指令：2 周期
   - 除法指令：可变周期

2. 内存延迟
   - 访问内存的时间
   - L1 缓存：1-2 周期
   - L2 缓存：10-20 周期
   - 主内存：100+ 周期

3. 端到端延迟
   - 整个任务的完成时间
   - 例如：一次推理从输入到输出的时间
```

**吞吐量 vs 延迟**：
```
类比：高速公路

吞吐量：
- 每小时通过多少辆车
- 可以通过增加车道提高

延迟：
- 一辆车从起点到终点需要多长时间
- 只能通过提高车速降低

关键区别：
- 高吞吐量不一定低延迟
- 低延迟不一定高吞吐量
- 需要根据应用场景权衡
```

### 11.1.3 CoralNPU 的性能特点

CoralNPU 是一个**高性能、低功耗**的 AI 加速处理器，具有以下性能特点：

#### 多发射架构

```
特性：每周期最多派发 4 条指令

优势：
- 理论峰值 IPC = 4.0
- 适合指令级并行度高的程序
- 向量指令可以充分利用多发射

限制：
- 必须是连续的 4 条指令
- 不能跳过中间有依赖的指令
- 实际 IPC 通常在 1.5-3.0 之间
```

#### 三级流水线

```
流水线阶段：
1. 取指（Fetch）：从内存取指令
2. 译码/派发（Decode/Dispatch）：解析指令并派发
3. 执行/写回（Execute/Writeback）：执行并写回结果

优势：
- 流水线级数少，分支预测失败代价小
- 时钟频率可以做得较高
- 设计简单，易于验证

权衡：
- 不如深流水线那样高频
- 但分支惩罚更小
```

#### 指令延迟

不同类型的指令有不同的执行延迟：

| 指令类型 | 延迟（周期） | 说明 | 示例 |
|---------|-------------|------|------|
| ALU | 1 | 算术逻辑运算 | add, sub, xor, and |
| CSR | 1 | 控制状态寄存器 | csrr, csrw |
| BRU | 1 | 分支跳转 | beq, jal, jalr |
| MLU | 2 | 乘法运算 | mul, mulh, mulhu |
| DVU | 可变 | 除法运算 | div, rem |
| LSU | 2+ | 访存操作 | lw, sw, lb, sb |

**延迟说明**：
```
1 周期指令（ALU/CSR/BRU）：
- 在执行阶段的第 1 个周期就完成
- 结果可以立即被后续指令使用
- 例如：add x1, x2, x3  # 1 周期后 x1 可用

2 周期指令（MLU）：
- 需要 2 个周期才能完成
- 后续依赖指令需要等待
- 例如：mul x1, x2, x3  # 2 周期后 x1 可用

可变周期指令（DVU）：
- 除法和取余运算
- 根据操作数不同，延迟不同
- 通常需要 10-30 个周期

2+ 周期指令（LSU）：
- 访存延迟取决于缓存命中情况
- L1 缓存命中：2-3 周期
- L2 缓存命中：10-20 周期
- 缓存未命中：100+ 周期
```

#### 向量扩展性能

CoralNPU 支持 RISC-V 向量扩展（RVV），具有以下性能特点：

```
向量长度：
- VLEN = 256 位
- 可以同时处理多个数据元素
- 例如：8 个 32 位整数，或 32 个 8 位整数

向量吞吐量：
- 向量加法：每周期 1 条向量指令
- 向量乘法：每周期 1 条向量指令
- 向量访存：每周期 1 条向量指令

性能提升：
- 相比标量代码，向量代码可以提升 4-8 倍性能
- 适合数据并行的应用（图像处理、矩阵运算）
```

#### 性能对比

与其他处理器的性能对比：

| 处理器 | 架构 | 峰值 IPC | 流水线级数 | 特点 |
|--------|------|---------|-----------|------|
| CoralNPU | 顺序多发 | 4.0 | 3 | 高 IPC，低延迟 |
| ARM Cortex-M7 | 顺序单发 | 1.0 | 6 | 低功耗 |
| ARM Cortex-A53 | 顺序双发 | 2.0 | 8 | 平衡性能功耗 |
| RISC-V Rocket | 顺序单发 | 1.0 | 5 | 简单高效 |
| RISC-V BOOM | 乱序多发 | 4.0+ | 7+ | 高性能 |

**CoralNPU 的定位**：
- 比简单的嵌入式处理器（Cortex-M）性能高
- 比复杂的乱序处理器（BOOM）功耗低
- 适合边缘 AI 推理场景

## 11.2 基准测试

基准测试（Benchmark）是评估处理器性能的标准化测试程序。通过运行这些测试，我们可以客观地比较不同处理器的性能。

### 11.2.1 什么是基准测试

**基准测试的目的**：
```
1. 性能评估
   - 测量处理器的实际性能
   - 发现性能瓶颈

2. 性能对比
   - 与其他处理器比较
   - 与上一代产品比较

3. 性能优化
   - 验证优化效果
   - 指导设计改进
```

**基准测试的类型**：
```
1. 综合基准测试
   - 测试多种计算能力
   - 例如：CoreMark, Dhrystone
   - 给出一个综合分数

2. 专项基准测试
   - 测试特定领域性能
   - 例如：ML 推理、图像处理
   - 更贴近实际应用

3. 微基准测试
   - 测试单个指令或操作
   - 例如：加法延迟、访存带宽
   - 用于深入分析
```

### 11.2.2 CoreMark

**CoreMark 简介**：

CoreMark 是由 EEMBC（嵌入式微处理器基准测试联盟）开发的处理器基准测试，专门用于测试嵌入式处理器的性能。

**CoreMark 测试内容**：
```
1. 列表处理（List Processing）
   - 链表的查找、排序、插入、删除
   - 测试指针操作和分支预测

2. 矩阵运算（Matrix Operations）
   - 矩阵乘法、矩阵加法
   - 测试算术运算能力

3. 状态机（State Machine）
   - 有限状态机的状态转换
   - 测试控制流和分支

4. CRC 校验（CRC）
   - 循环冗余校验计算
   - 测试位操作和循环
```

**CoreMark 评分**：
```
CoreMark 分数 = 每秒迭代次数

例如：
- CoreMark 分数 = 1000
- 表示每秒可以完成 1000 次 CoreMark 测试迭代
```

**CoralNPU 的 CoreMark 性能**：

```
测试配置：
- 时钟频率：100 MHz
- 编译器：GCC 12.2.0
- 优化级别：-O3
- 内存：TCM（紧耦合内存）

测试结果：
- CoreMark 分数：约 350-400 CoreMark
- CoreMark/MHz：3.5-4.0
- IPC：约 2.5-3.0

性能分析：
- 多发射架构充分利用了指令级并行
- 紧耦合内存减少了访存延迟
- 分支预测提高了控制流性能
```

**性能对比**：

| 处理器 | 频率 | CoreMark | CoreMark/MHz |
|--------|------|----------|--------------|
| CoralNPU | 100 MHz | 350-400 | 3.5-4.0 |
| ARM Cortex-M7 | 200 MHz | 400-500 | 2.0-2.5 |
| ARM Cortex-A53 | 1 GHz | 2500-3000 | 2.5-3.0 |
| RISC-V Rocket | 100 MHz | 150-200 | 1.5-2.0 |

**结论**：
- CoralNPU 的 CoreMark/MHz 性能优秀
- 多发射架构带来了显著的性能提升
- 在相同频率下，性能是简单单发射处理器的 2-3 倍

### 11.2.3 Dhrystone

**Dhrystone 简介**：

Dhrystone 是一个更古老但仍然广泛使用的基准测试，主要测试整数运算性能。

**Dhrystone 测试内容**：
```
1. 字符串操作
   - 字符串复制、比较
   - 测试内存访问和字符处理

2. 整数运算
   - 加减乘除、位运算
   - 测试 ALU 性能

3. 控制流
   - if-else、switch-case
   - 测试分支性能

4. 函数调用
   - 函数调用和返回
   - 测试栈操作和跳转
```

**Dhrystone 评分**：
```
Dhrystone 分数 = DMIPS (Dhrystone MIPS)

DMIPS = (Dhrystone 每秒迭代次数) / 1757

例如：
- 每秒完成 100,000 次迭代
- DMIPS = 100,000 / 1757 ≈ 56.9
```

**为什么除以 1757**：
- 这是 VAX 11/780 计算机的 Dhrystone 性能
- VAX 11/780 被定义为 1 MIPS
- 所以 1757 次/秒 = 1 MIPS

**CoralNPU 的 Dhrystone 性能**：

```
测试配置：
- 时钟频率：100 MHz
- 编译器：GCC 12.2.0
- 优化级别：-O3

测试结果：
- Dhrystone 分数：约 150-180 DMIPS
- DMIPS/MHz：1.5-1.8
- IPC：约 2.0-2.5

性能分析：
- 整数运算性能优秀
- 函数调用开销较小
- 内存访问效率高
```

## 11.3 性能分析和优化

性能分析是找出程序性能瓶颈的过程，性能优化是改进程序使其运行更快的过程。

### 11.3.1 性能计数器

**什么是性能计数器**：

性能计数器（Performance Counter）是处理器内部的硬件计数器，用于统计各种性能事件。

**CoralNPU 的性能计数器**：

CoralNPU 实现了 RISC-V 标准的性能计数器：

| 计数器 | CSR 地址 | 说明 |
|--------|---------|------|
| mcycle | 0xB00 | 时钟周期计数器（低 32 位） |
| mcycleh | 0xB80 | 时钟周期计数器（高 32 位） |
| minstret | 0xB02 | 指令退休计数器（低 32 位） |
| minstreth | 0xB82 | 指令退休计数器（高 32 位） |

**计数器说明**：
```
mcycle/mcycleh：
- 统计从处理器启动以来经过的时钟周期数
- 64 位计数器，分为高 32 位和低 32 位
- 每个时钟周期自动加 1
- 可以用来测量时间

minstret/minstreth：
- 统计从处理器启动以来退休（完成）的指令数
- 64 位计数器，分为高 32 位和低 32 位
- 每条指令完成时加 1
- 可以用来统计指令数

IPC 计算：
IPC = minstret / mcycle
```

**为什么需要 64 位计数器**：
```
32 位计数器的问题：
- 假设时钟频率 100 MHz
- 32 位计数器最大值：2^32 = 4,294,967,296
- 溢出时间：4,294,967,296 / 100,000,000 ≈ 43 秒

64 位计数器：
- 最大值：2^64 = 18,446,744,073,709,551,616
- 溢出时间：约 5800 年
- 实际应用中不会溢出
```

### 11.3.2 如何使用性能计数器

**读取性能计数器**：

在 C 语言中，可以使用内联汇编读取性能计数器：

```c
#include <stdint.h>

// 读取 mcycle 计数器（64 位）
uint64_t read_mcycle(void) {
    uint32_t cycle_low = 0;
    uint32_t cycle_high = 0;
    uint32_t cycle_high_2 = 0;
    
    // 读取顺序很重要，防止溢出导致错误
    asm volatile(
        "1:"
        "  csrr %0, mcycleh;"  // 先读高位
        "  csrr %1, mcycle;"   // 再读低位
        "  csrr %2, mcycleh;"  // 再次读高位
        "  bne  %0, %2, 1b;"   // 如果两次高位不同，说明发生了溢出，重新读取
        : "=r"(cycle_high), "=r"(cycle_low), "=r"(cycle_high_2)
    );
    
    return ((uint64_t)cycle_high << 32) | cycle_low;
}

// 读取 minstret 计数器（64 位）
uint64_t read_minstret(void) {
    uint32_t instret_low = 0;
    uint32_t instret_high = 0;
    uint32_t instret_high_2 = 0;
    
    asm volatile(
        "1:"
        "  csrr %0, minstreth;"
        "  csrr %1, minstret;"
        "  csrr %2, minstreth;"
        "  bne  %0, %2, 1b;"
        : "=r"(instret_high), "=r"(instret_low), "=r"(instret_high_2)
    );
    
    return ((uint64_t)instret_high << 32) | instret_low;
}
```

**为什么读取顺序很重要**：
```
问题场景：
1. 读取 mcycleh = 0x00000001
2. 此时 mcycle 从 0xFFFFFFFF 溢出到 0x00000000
3. 读取 mcycle = 0x00000000
4. mcycleh 自动加 1，变成 0x00000002
5. 结果：0x0000000100000000（错误！）
6. 正确值应该是：0x0000000200000000

解决方法：
1. 读取 mcycleh（第一次）
2. 读取 mcycle
3. 读取 mcycleh（第二次）
4. 如果两次 mcycleh 不同，说明发生了溢出，重新读取
5. 这样可以保证读取的值是一致的
```

**测量代码性能**：

```c
#include <stdio.h>

void test_function(void) {
    // 要测试的代码
    int sum = 0;
    for (int i = 0; i < 1000; i++) {
        sum += i;
    }
}

int main(void) {
```

### 11.3.3 性能瓶颈分析

**常见性能瓶颈**：

#### 1. 数据依赖

**问题**：后面的指令需要等待前面指令的结果。

```c
// 示例：数据依赖导致的性能问题
int a = b + c;      // 指令 1
int d = a + e;      // 指令 2：依赖指令 1 的结果
int f = d + g;      // 指令 3：依赖指令 2 的结果
int h = f + i;      // 指令 4：依赖指令 3 的结果

// 性能分析：
// - 每条指令都依赖前一条指令
// - 无法并行执行
// - IPC ≈ 1.0（即使处理器支持 4 发射）
```

**解决方法**：打破依赖链

```c
// 优化后：减少依赖
int a = b + c;      // 指令 1
int d = e + f;      // 指令 2：独立于指令 1
int g = h + i;      // 指令 3：独立于指令 1 和 2
int j = k + l;      // 指令 4：独立于前面所有指令

int result = a + d + g + j;  // 最后合并结果

// 性能分析：
// - 前 4 条指令可以并行执行
// - IPC ≈ 4.0（充分利用 4 发射）
// - 性能提升 4 倍
```

#### 2. 访存延迟

**问题**：访问内存的延迟远大于计算延迟。

```c
// 示例：频繁访存导致的性能问题
int sum = 0;
for (int i = 0; i < 1000; i++) {
    sum += array[i];  // 每次循环都要访问内存
}

// 性能分析：
// - 每次循环需要 1 次内存访问（2+ 周期）
// - 加法只需要 1 周期
// - 访存成为瓶颈
// - IPC ≈ 0.5（大部分时间在等待内存）
```

**解决方法**：减少访存次数

```c
// 优化 1：循环展开
int sum = 0;
for (int i = 0; i < 1000; i += 4) {
    sum += array[i];
    sum += array[i+1];
    sum += array[i+2];
    sum += array[i+3];
}

// 优化 2：使用向量指令
// 向量指令可以一次加载多个数据
// 减少访存次数，提高性能
```

#### 3. 分支预测失败

**问题**：分支指令导致流水线停顿。

```c
// 示例：难以预测的分支
int sum = 0;
for (int i = 0; i < 1000; i++) {
    if (array[i] > threshold) {  // 分支难以预测
        sum += array[i];
    }
}

// 性能分析：
// - 如果 array[i] > threshold 的概率是 50%
// - 分支预测准确率约 50%
// - 每次预测失败需要清空流水线（3 周期惩罚）
// - 性能损失约 50% * 3 = 1.5 周期/迭代
```

**解决方法**：消除分支

```c
// 优化：使用条件移动指令
int sum = 0;
for (int i = 0; i < 1000; i++) {
    int mask = (array[i] > threshold) ? 1 : 0;
    sum += array[i] * mask;
}

// 或者使用 RISC-V 的条件移动指令（伪代码）
// cmov rd, rs1, rs2, condition
// 如果 condition 为真，rd = rs1，否则 rd = rs2
```

#### 4. 资源冲突

**问题**：多条指令竞争同一个硬件单元。

```c
// 示例：多条乘法指令
int a = b * c;      // 乘法 1
int d = e * f;      // 乘法 2
int g = h * i;      // 乘法 3
int j = k * l;      // 乘法 4

// 性能分析：
// - 如果只有 1 个乘法器
// - 4 条乘法指令需要串行执行
// - 即使处理器支持 4 发射，也无法并行
// - IPC ≈ 1.0
```

**解决方法**：混合不同类型的指令

```c
// 优化：混合乘法和加法
int a = b * c;      // 乘法 1
int d = e + f;      // 加法 1（可以并行）
int g = h * i;      // 乘法 2
int j = k + l;      // 加法 2（可以并行）

// 性能分析：
// - 乘法和加法使用不同的执行单元
// - 可以并行执行
// - IPC ≈ 2.0-3.0
```

## 11.4 功耗分析

功耗（Power Consumption）是处理器消耗的电能，对于边缘设备和嵌入式系统来说，功耗是一个非常重要的指标。

### 11.4.1 什么是功耗

**功耗的基本概念**：

```
功耗 = 电压 × 电流

单位：
- 瓦特（W）= 伏特（V） × 安培（A）
- 毫瓦（mW）= 0.001 W

例如：
- 电压：1.0 V
- 电流：50 mA = 0.05 A
- 功耗：1.0 V × 0.05 A = 0.05 W = 50 mW
```

**功耗的类型**：

```
1. 动态功耗（Dynamic Power）
   - 电路翻转时消耗的能量
   - 与时钟频率和活动率成正比
   - 公式：P_dynamic = α × C × V^2 × f
     - α：活动因子（0-1）
     - C：电容
     - V：电压
     - f：频率

2. 静态功耗（Static Power）
   - 电路静止时的漏电流
   - 与温度和工艺相关
   - 公式：P_static = V × I_leakage

3. 总功耗
   - P_total = P_dynamic + P_static
```

**类比理解**：
```
动态功耗：
- 就像汽车行驶时消耗的燃料
- 速度越快，消耗越多
- 停车时不消耗

静态功耗：
- 就像汽车怠速时消耗的燃料
- 即使不动也在消耗
- 温度越高，消耗越多
```

### 11.4.2 CoralNPU 的功耗特性

**功耗组成**：

```
CoralNPU 总功耗 = 核心功耗 + 外设功耗 + 内存功耗

核心功耗：
- 标量流水线：20-30 mW @ 100 MHz
- 向量单元：30-50 mW @ 100 MHz（活动时）
- 控制逻辑：5-10 mW

外设功耗：
- UART：1-2 mW
- SPI：2-5 mW
- I2C：1-2 mW
- GPIO：< 1 mW

内存功耗：
- TCM：10-20 mW
- 缓存：5-10 mW
```

**典型功耗数据**：

| 工作模式 | 频率 | 功耗 | 说明 |
|---------|------|------|------|
| 空闲 | 100 MHz | 10-20 mW | 只有时钟和静态功耗 |
| 标量计算 | 100 MHz | 30-50 mW | 运行标量程序 |
| 向量计算 | 100 MHz | 60-100 mW | 运行向量程序 |
| ML 推理 | 100 MHz | 80-120 mW | 运行 ML 模型 |
| 峰值 | 100 MHz | 100-150 mW | 所有单元全速运行 |

**功耗与频率的关系**：

```
动态功耗与频率成正比：
- 50 MHz：30-50 mW
- 100 MHz：60-100 mW
- 200 MHz：120-200 mW

能效（性能/功耗）：
- 50 MHz：较低性能，最佳能效
- 100 MHz：平衡性能和功耗
- 200 MHz：最高性能，较高功耗
```

### 11.4.3 功耗测量

**测量方法**：

#### 方法 1：硬件测量

```
使用电流表测量：
1. 将电流表串联在电源和芯片之间
2. 测量电流 I
3. 已知电压 V
4. 计算功耗：P = V × I

优点：
- 准确
- 实时

缺点：
- 需要硬件设备
- 无法区分不同模块的功耗
```

#### 方法 2：仿真估算

```
使用 EDA 工具估算：
1. 综合设计，生成网表
2. 运行功耗分析工具（如 PrimeTime PX）
3. 输入活动率信息
4. 工具计算功耗

优点：
- 可以分析各个模块的功耗
- 设计阶段就能评估

缺点：
- 需要准确的活动率信息
- 估算值可能与实际有偏差
```

#### 方法 3：性能计数器估算

```c
// 使用性能计数器估算功耗
uint64_t start_cycle = read_mcycle();
uint64_t start_instret = read_minstret();

// 运行测试程序
test_function();

uint64_t end_cycle = read_mcycle();
uint64_t end_instret = read_minstret();

// 计算活动率
double activity = (double)(end_instret - start_instret) / 
                  (double)(end_cycle - start_cycle);

// 估算功耗（经验公式）
double power_mw = 20 + activity * 80;  // 20 mW 基础 + 80 mW 峰值

printf("Estimated power: %.2f mW\n", power_mw);
```

### 11.4.4 低功耗模式

CoralNPU 支持多种低功耗模式，以适应不同的应用场景。

#### 模式 1：时钟门控（Clock Gating）

**原理**：关闭不使用的模块的时钟。

```
实现：
- 硬件自动检测模块是否空闲
- 空闲时关闭时钟
- 需要时自动恢复

功耗节省：
- 可以节省 30-50% 的动态功耗
- 对性能无影响

适用场景：
- 所有场景
- 默认启用
```

#### 模式 2：动态电压频率调节（DVFS）

**原理**：根据负载动态调整电压和频率。

```
工作模式：
- 高性能：1.0 V, 200 MHz, 150 mW
- 平衡：0.9 V, 100 MHz, 60 mW
- 低功耗：0.8 V, 50 MHz, 20 mW

切换方式：
- 软件控制
- 根据任务需求切换

功耗节省：
- 可以节省 50-70% 的功耗
- 性能相应降低

适用场景：
- 对实时性要求不高的任务
- 电池供电设备
```

#### 模式 3：睡眠模式（Sleep Mode）

**原理**：关闭大部分电路，只保留必要的唤醒逻辑。

```
睡眠级别：
1. 浅睡眠（Light Sleep）
   - 关闭核心，保留内存
   - 功耗：5-10 mW
   - 唤醒时间：< 1 ms

2. 深睡眠（Deep Sleep）
   - 关闭核心和大部分内存
   - 功耗：1-2 mW
   - 唤醒时间：1-10 ms

3. 关机（Power Off）
   - 关闭所有电路
   - 功耗：< 0.1 mW
   - 唤醒时间：> 100 ms

使用方法：
// 进入睡眠模式
asm volatile("wfi");  // Wait For Interrupt

// 被中断唤醒后继续执行
```

**睡眠模式示例**：

```c
#include <stdint.h>

// 配置唤醒源（例如：定时器中断）
void setup_wakeup_timer(uint32_t ms) {
    // 配置定时器在 ms 毫秒后产生中断
    // 具体实现取决于硬件
}

// 进入睡眠模式
void enter_sleep_mode(void) {
    // 1. 保存必要的状态
    // 2. 关闭不需要的外设
    // 3. 进入睡眠
    asm volatile("wfi");
    // 4. 被唤醒后恢复状态
}

int main(void) {
    while (1) {
        // 执行任务
        do_work();
        
        // 如果没有任务，进入睡眠
        if (no_more_work()) {
            setup_wakeup_timer(1000);  // 1 秒后唤醒
            enter_sleep_mode();
        }
    }
    return 0;
}
```

#### 策略 1：降低时钟频率

```
- 降低电压可以大幅降低功耗
```

#### 策略 4：优化算法

```c
// 原始算法：计算密集
void process_data_naive(uint8_t* data, int n) {
    for (int i = 0; i < n; i++) {
        // 复杂的浮点运算
        float result = sqrt(data[i]) * log(data[i]);
        data[i] = (uint8_t)result;
    }
}
// 功耗：100 mW，执行时间：100 ms

// 优化算法：查表法
uint8_t lookup_table[256];  // 预计算的查找表

void process_data_optimized(uint8_t* data, int n) {
    for (int i = 0; i < n; i++) {
        data[i] = lookup_table[data[i]];
    }
}
// 功耗：50 mW，执行时间：10 ms
// 总能量：从 10 mJ 降到 0.5 mJ（20 倍节省）
```

#### 策略 5：批处理

```c
// 原始方案：来一个处理一个
void process_streaming(void) {
    while (1) {
        if (has_data()) {
            uint8_t data = read_data();
            process(data);
        } else {
            // 空闲等待，浪费功耗
        }
    }
}

// 优化方案：批量处理
void process_batched(void) {
    while (1) {
        // 收集一批数据
        uint8_t batch[100];
        int count = 0;
        while (count < 100 && has_data()) {
            batch[count++] = read_data();
        }
        
        // 批量处理
        if (count > 0) {
            process_batch(batch, count);
        }
        
        // 没有数据时进入睡眠
        if (!has_data()) {
            enter_sleep_mode();
        }
    }
}

// 优势：
// - 减少唤醒次数
// - 提高缓存利用率
// - 功耗节省 30-50%
```

## 11.5 面积分析

面积（Area）是指芯片上电路占用的硅片面积，对于芯片设计来说，面积直接影响成本。

### 11.5.1 什么是芯片面积

**面积的基本概念**：

```
芯片面积 = 长 × 宽

单位：
- mm² (平方毫米)
- 例如：5 mm × 5 mm = 25 mm²

面积的组成：
1. 逻辑电路面积
   - 处理器核心
   - 控制逻辑
   - 执行单元

2. 存储器面积
   - 寄存器文件
   - 缓存
   - TCM

3. 互连面积
   - 连线
   - 总线

4. 其他
   - 时钟树
   - 电源网络
   - 填充单元
```

**面积与成本的关系**：

```
芯片成本 ≈ 面积 × 单位面积成本

例如：
- 芯片面积：25 mm²
- 单位面积成本：$0.10/mm²（28nm 工艺）
- 芯片成本：25 × 0.10 = $2.50

关键点：
- 面积越大，成本越高
- 面积越大，良率越低（更容易有缺陷）
- 需要在性能和面积之间权衡
```

**类比理解**：
```
芯片面积就像房子的占地面积：
- 面积越大，成本越高
- 面积越大，可以放更多功能
- 需要合理规划，避免浪费
```

## 11.6 性能总结
