# 微架构

## 微架构概述

### 什么是微架构

**微架构（Microarchitecture）** 是指处理器内部的具体实现方式，它描述了指令在硬件中如何被执行。

**架构 vs 微架构的区别**：

```
架构（Architecture）：
  - 程序员可见的接口
  - 定义：指令集、寄存器、内存模型
  - 例子：RISC-V RV32I 定义了 32 个通用寄存器、ADD/SUB 等指令
  - 特点：抽象的、不变的规范

微架构（Microarchitecture）：
  - 硬件实现的细节
  - 定义：流水线级数、执行单元数量、缓存大小
  - 例子：CoralNPU 使用 3 级流水线、4 个 ALU
  - 特点：具体的、可变的实现
```

**类比理解**：
- **架构**就像是一份菜谱，规定了做一道菜需要哪些食材和步骤
- **微架构**就像是厨房的布局和设备，决定了如何高效地完成这道菜

### CoralNPU 微架构特点

CoralNPU 采用经典的**顺序多发、乱序完成、顺序提交**架构：

```
顺序多发（In-Order Multi-Issue）：
  - 按照程序顺序派发指令
  - 每周期可以派发多条指令（最多 4 条）
  - 必须连续派发（不能跳过中间的指令）

乱序完成（Out-of-Order Completion）：
  - 指令可以不按派发顺序完成
  - 短延迟指令可以先完成
  - 长延迟指令慢慢执行

顺序提交（In-Order Commit/Retirement）：
  - 指令必须按程序顺序写回架构状态
  - 保证程序语义正确性
  - 实现精确异常
```

**设计权衡**：

| 设计选择 | 目的 | 收益 | 代价 |
|---------|------|------|------|
| 顺序多发 | 简化控制逻辑 | 硬件简单，易于验证 | 可能因依赖暂停 |
| 乱序完成 | 隐藏长延迟 | 性能提升 50-150% | 需要记分牌逻辑 |
| 顺序提交 | 保证正确性 | 精确异常，可调试 | 需要额外控制逻辑 |

## 流水线设计

CoralNPU 标量核心采用**三级流水线**设计：

```
流水线阶段：

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   取指      │ --> │ 译码/派发   │ --> │ 执行/写回   │
│   Fetch     │     │Decode/Disp. │     │Execute/WB   │
└─────────────┘     └─────────────┘     └─────────────┘
     1 周期              1 周期              1+ 周期
```

详细的流水线阶段说明请参见第三章第二节《标量核心》。

## 指令派发机制

### 顺序多发原理

**核心思想**：按照程序顺序尝试派发指令，直到遇到无法派发的指令为止。

**lastReady 链机制**：

```scala
// 伪代码
val lastReady = Wire(Vec(5, Bool()))
lastReady(0) := true.B  // 第一个槽总是可以尝试

for (i <- 0 until 4) {
  val tryDispatch = lastReady(i) && canDispatch(i)
  
  // 尝试派发到执行单元
  val dispatched = 派发成功?
  
  lastReady(i + 1) := dispatched  // 传递给下一个槽
}
```

**工作原理**：

```
示例：4 条指令

指令 0: ADD x1, x2, x3   (无依赖)
指令 1: DIV x4, x5, x6   (无依赖)
指令 2: MUL x7, x1, x8   (依赖 x1，指令 0 写)
指令 3: ADD x9, x10, x11 (无依赖)

派发过程：

lastReady(0) = true
  ↓
指令 0 尝试派发 -> 成功 ✓
  ↓
lastReady(1) = true
  ↓
指令 1 尝试派发 -> 成功 ✓
  ↓
lastReady(2) = true
  ↓
指令 2 尝试派发 -> 失败 ✗ (RAW 冒险，x1 忙)
  ↓
lastReady(3) = false
  ↓
指令 3 不尝试派发

结果：派发指令 0, 1（连续，到冒险为止）
```

### 记分牌机制（Scoreboard）

**什么是记分牌**：
记分牌是一个 32 位的位向量（bit vector），每一位对应一个寄存器。
- `scoreboard[i] = 1`：寄存器 xi 有待完成的写入（不能读）
- `scoreboard[i] = 0`：寄存器 xi 可以安全读取

**记分牌操作**：

```
设置（派发时）：
  当指令派发时，将其目标寄存器对应的位置 1
  scoreboard |= (1 << rd_addr)

清除（写回时）：
  当指令写回时，将其目标寄存器对应的位清 0
  scoreboard &= ~(1 << rd_addr)

检查（派发时）：
  检查源寄存器对应的位是否为 1
  if (scoreboard & (1 << rs1_addr)) != 0:
      -> RAW 冒险，暂停
```

**ASCII 图示**：

```
记分牌状态示例：

寄存器:  x0  x1  x2  x3  x4  x5  ...  x31
记分牌:  0   1   0   0   1   0  ...   0
         ↑   ↑           ↑
         |   |           |
      总是0  忙         忙

解释：
  - x0 总是 0（RISC-V 规定）
  - x1 有待写入（某条指令正在计算结果）
  - x4 有待写入
  - 其他寄存器可以安全读取
```

### 依赖检查

**三种数据冒险**：

1. **RAW（Read After Write）冒险**：
   - 后面的指令读取前面指令要写的寄存器
   - 例子：
     ```
     指令 1: ADD x1, x2, x3   (写 x1)
     指令 2: SUB x4, x1, x5   (读 x1)  <- RAW 冒险
     ```
   - 解决：检查记分牌，如果 x1 忙则暂停指令 2

2. **WAW（Write After Write）冒险**：
   - 两条指令写同一个寄存器
   - 例子：
     ```
     指令 1: ADD x1, x2, x3   (写 x1)
     指令 2: SUB x1, x4, x5   (写 x1)  <- WAW 冒险
     ```
   - 解决：检查记分牌，如果 x1 忙则暂停指令 2

3. **WAR（Write After Read）冒险**：
   - 后面的指令写前面指令要读的寄存器
   - 例子：
     ```
     指令 1: ADD x3, x1, x2   (读 x1)
     指令 2: SUB x1, x4, x5   (写 x1)  <- WAR 冒险
     ```
   - **CoralNPU 中不会发生**：因为所有指令在派发后的下一个周期就读取操作数

**为什么 CoralNPU 不会发生 WAR 冒险**：

```
时间线：

周期 1: 指令 1, 2 派发
周期 2: 指令 1 读 x1, x2（从寄存器文件）
        指令 2 读 x4, x5（从寄存器文件）
周期 3: 指令 1 写 x3
        指令 2 写 x1

关键：所有指令在同一周期读取操作数，
     写回在后面的周期，不会冲突
```

## 乱序执行

### 为什么需要乱序执行

**问题**：不同指令的执行延迟不同

```
示例程序：

指令 0: ADD x1, x2, x3   (ALU, 1 周期)
指令 1: DIV x4, x5, x6   (DVU, 32 周期)
指令 2: MUL x7, x8, x9   (MLU, 2 周期)
指令 3: ADD x10, x11, x12 (ALU, 1 周期)

如果顺序执行：
  周期 1: 指令 0 执行
  周期 2: 指令 0 完成，指令 1 开始
  周期 33: 指令 1 完成，指令 2 开始
  周期 35: 指令 2 完成，指令 3 开始
  周期 36: 指令 3 完成
  总时间：36 周期

如果乱序执行：
  周期 1: 指令 0, 1 派发
  周期 2: 指令 0 完成，指令 2, 3 派发
  周期 3: 指令 3 完成
  周期 4: 指令 2 完成
  周期 33: 指令 1 完成
  总时间：33 周期

加速：约 9%
```

**乱序执行的本质**：
- 让快的指令先完成，不等慢的指令
- 充分利用多个执行单元的并行性
- 隐藏长延迟操作（除法、访存）

### 乱序完成的原因

**原因 1：不同的执行延迟**

```
执行单元延迟对比：

ALU:  ████ 1 周期
BRU:  ████ 1 周期
MLU:  ████████ 2 周期
DVU:  ████████████████████████████████ 32 周期
LSU:  ████████ 2+ 周期（取决于内存）

同时派发的指令会在不同时间完成
```

**原因 2：并行执行**

```
多个执行单元同时工作：

时间 ->
ALU(0) ━━ 指令0 ━━ 完成
ALU(1) ━━━━━━━━━━━━━━  (空闲)
ALU(2) ━━━━━━━━━━━━━━  (空闲)
ALU(3) ━━ 指令3 ━━ 完成
MLU    ━━━━ 指令2 ━━━━ 完成
DVU    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 指令1 ━━ 完成

完成顺序：0 -> 3 -> 2 -> 1
派发顺序：0 -> 1 -> 2 -> 3
```

## 执行单元仲裁

### 共享执行单元的仲裁

CoralNPU 有 4 个指令槽，但某些执行单元只有 1 个（如 MLU、DVU、LSU）。当多条指令同时需要同一个执行单元时，需要仲裁。

**仲裁策略**：优先级仲裁（Priority Arbitration）

```
优先级：槽 0 > 槽 1 > 槽 2 > 槽 3

示例：
  槽 0: MUL x1, x2, x3  (需要 MLU)
  槽 1: MUL x4, x5, x6  (需要 MLU)
  槽 2: ADD x7, x8, x9  (需要 ALU)
  槽 3: MUL x10, x11, x12 (需要 MLU)

仲裁结果：
  - 槽 0 获得 MLU（优先级最高）
  - 槽 1 暂停（MLU 被占用）
  - 槽 2 获得 ALU(2)（无冲突）
  - 槽 3 不尝试（lastReady 链断开）

本周期派发：槽 0, 1 暂停，槽 2 派发，槽 3 不尝试
```

**仲裁逻辑**：

```scala
// 伪代码
val mluRequest = Vec(4, Bool())  // 4 个槽是否请求 MLU
val mluGrant = Wire(Vec(4, Bool()))

// 优先级编码器
mluGrant(0) := mluRequest(0)
mluGrant(1) := mluRequest(1) && !mluRequest(0)
mluGrant(2) := mluRequest(2) && !mluRequest(1) && !mluRequest(0)
mluGrant(3) := mluRequest(3) && !mluRequest(2) && !mluRequest(1) && !mluRequest(0)
```

## 写回和提交

### 写回阶段

**写回端口配置**：

```
寄存器文件写端口：
  - ALU 写端口：4 个（每个槽一个）
  - MLU 写端口：1 个（共享）
  - DVU 写端口：1 个（共享）
  - LSU 写端口：1 个（共享）
  - FPU 写端口：1 个（共享）

总计：最多 8 个写端口（但实际同时使用的少于 8 个）
```

**写回冲突处理**：

由于 CoralNPU 采用顺序提交，写回时不会有冲突：
- 每条指令有唯一的目标寄存器
- 记分牌保证不会有 WAW 冒险
- 写回可以乱序进行

### 提交阶段

**什么是提交（Commit/Retirement）**：

提交是指将指令的结果永久写入架构状态（寄存器文件、内存）。在 CoralNPU 中，写回和提交是同时进行的，因为采用顺序提交策略。

**提交条件**：

```
指令可以提交当且仅当：
  1. 指令已经完成执行
  2. 指令的结果已经写回寄存器文件
  3. 所有前面的指令都已经提交（顺序提交）
```

**精确异常（Precise Exception）**：

顺序提交保证了精确异常：
- 异常发生时，所有前面的指令都已完成
- 所有后面的指令都未开始
- 处理器状态与顺序执行一致

## 性能分析

### 理论性能

**理想情况**（无依赖、无冲突）：

```
每周期派发 4 条指令
IPC (Instructions Per Cycle) = 4
```

**实际性能**（有依赖、有冲突）：

```
典型 ML 推理代码：
  - 标量控制代码：IPC ≈ 1.5-2.0
  - 向量计算代码：IPC ≈ 0.5-1.0（受向量执行延迟限制）
  - 混合代码：IPC ≈ 1.0-1.5
```

### 性能瓶颈

**瓶颈 1：数据依赖**

```
依赖链示例：

ADD x1, x2, x3   (1 周期)
  ↓ (x1 依赖)
MUL x4, x1, x5   (2 周期)
  ↓ (x4 依赖)
ADD x6, x4, x7   (1 周期)

总延迟：1 + 2 + 1 = 4 周期
无法并行
```

**瓶颈 2：结构冒险**

```
多条指令竞争同一执行单元：

MUL x1, x2, x3   (需要 MLU)
MUL x4, x5, x6   (需要 MLU) <- 暂停
MUL x7, x8, x9   (需要 MLU) <- 暂停
```

**瓶颈 3：控制流**

```
分支指令打断流水线：

BEQ x1, x2, label
  ↓ (分支预测)
ADD x3, x4, x5   (可能被刷新)
```

### 优化策略

**优化 1：循环展开**

```
原始循环：
  for (i = 0; i < 100; i++) {
    c[i] = a[i] + b[i];
  }

展开后：
  for (i = 0; i < 100; i += 4) {
    c[i+0] = a[i+0] + b[i+0];
    c[i+1] = a[i+1] + b[i+1];
    c[i+2] = a[i+2] + b[i+2];
    c[i+3] = a[i+3] + b[i+3];
  }

好处：
  - 减少分支指令数量
  - 增加指令级并行性
  - 4 条 ADD 可以同时派发
```

**优化 2：软件流水线**

```
重排指令以减少依赖：

原始顺序：
  LW x1, 0(x10)    (加载)
  ADD x2, x1, x3   (依赖 x1)
  SW x2, 0(x11)    (存储)

优化后：
  LW x1, 0(x10)    (加载)
  LW x4, 4(x10)    (加载下一个，无依赖)
  ADD x2, x1, x3   (依赖 x1，但 x4 加载可以并行)
  SW x2, 0(x11)    (存储)
```

**优化 3：使用向量指令**

```
标量代码：
  for (i = 0; i < 8; i++) {
    c[i] = a[i] + b[i];
  }
  需要 8 次循环

向量代码：
  vle v0, (a)      # 加载 a[0:7]
  vle v1, (b)      # 加载 b[0:7]
  vadd v2, v0, v1  # c[0:7] = a[0:7] + b[0:7]
  vse v2, (c)      # 存储 c[0:7]
  只需 4 条指令
```

## 总结

CoralNPU 的微架构采用了经典的顺序多发、乱序完成、顺序提交设计，在简化硬件的同时提供了良好的性能：

- **顺序多发**：简化控制逻辑，易于验证
- **乱序完成**：隐藏长延迟操作，提高吞吐量
- **顺序提交**：保证精确异常，便于调试
- **记分牌机制**：高效的依赖检查
- **优先级仲裁**：公平分配共享资源

这种设计在性能、功耗和复杂度之间取得了良好的平衡，非常适合嵌入式 ML 推理应用。
