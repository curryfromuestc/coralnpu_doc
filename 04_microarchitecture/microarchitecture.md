# 第四章：微架构

## 4.1 微架构概述

### 4.1.1 什么是微架构

**微架构（Microarchitecture）** 是指处理器内部的具体实现方式，它描述了指令在硬件中如何被执行。

**架构 vs 微架构的区别**：

```
架构（Architecture）：
  - 程序员可见的接口
  - 定义：指令集、寄存器、内存模型
  - 例子：RISC-V RV32I 定义了 32 个通用寄存器、ADD/SUB 等指令
  - 特点：抽象的、不变的规范

微架构（Microarchitecture）：
  - 硬件实现的细节
  - 定义：流水线级数、执行单元数量、缓存大小
  - 例子：CoralNPU 使用 4 级流水线、4 个 ALU
  - 特点：具体的、可变的实现
```

**类比理解**：
- **架构**就像是一份菜谱，规定了做一道菜需要哪些食材和步骤
- **微架构**就像是厨房的布局和设备，决定了如何高效地完成这道菜

**为什么需要微架构**：
- 同一个架构可以有多种微架构实现
- 不同的微架构在性能、功耗、面积上有不同的权衡
- CoralNPU 的微架构设计目标是：高性能、低功耗、易验证

### 4.1.2 CoralNPU 微架构特点

CoralNPU 采用经典的**顺序多发、乱序完成、顺序提交**架构：

```
顺序多发（In-Order Multi-Issue）：
  - 按照程序顺序派发指令
  - 每周期可以派发多条指令（最多 4 条）
  - 必须连续派发（不能跳过中间的指令）

乱序完成（Out-of-Order Completion）：
  - 指令可以不按派发顺序完成
  - 短延迟指令可以先完成
  - 长延迟指令慢慢执行

顺序提交（In-Order Commit/Retirement）：
  - 指令必须按程序顺序写回架构状态
  - 保证程序语义正确性
  - 实现精确异常
```

**设计权衡**：

| 设计选择 | 目的 | 收益 | 代价 |
|---------|------|------|------|
| 顺序多发 | 简化控制逻辑 | 硬件简单，易于验证 | 可能因依赖暂停 |
| 乱序完成 | 隐藏长延迟 | 性能提升 50-150% | 需要 ROB 缓冲 |
| 顺序提交 | 保证正确性 | 精确异常，可调试 | ROB 硬件成本 |

## 4.2 流水线阶段

CoralNPU 标量核心采用**三级流水线**设计：

```
流水线阶段：

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   取指      │ --> │ 译码/派发   │ --> │ 执行/写回   │
│   Fetch     │     │Decode/Disp. │     │Execute/WB   │
└─────────────┘     └─────────────┘     └─────────────┘
     1 周期              1 周期              1+ 周期
```

### 4.2.1 取指阶段（Fetch）

**功能**：从内存中取出指令，放入指令缓冲区。

**特性**：
- 每周期取 4 条指令（256 位 = 32 字节）
- L0 指令缓存（1KB）减少访问延迟
- 集成分支预测（前向/后向预测）

**ASCII 图示**：

```
取指单元结构：

                    ┌──────────────┐
                    │   PC 寄存器  │
                    └──────┬───────┘
                           │
                           v
                    ┌──────────────┐
                    │ L0 指令缓存  │
                    │   (1KB)      │
                    └──────┬───────┘
                           │
                           v
                    ┌──────────────┐
                    │ 指令缓冲区   │
                    │ (4 条指令)   │
                    └──────┬───────┘
                           │
                           v
                    ┌──────────────┐
                    │ 分支预测器   │
                    └──────────────┘
```

**分支预测策略**：
- **后向分支**（地址减小）：预测为跳转（循环通常会执行多次）
- **前向分支**（地址增大）：预测为不跳转（if 语句通常不执行）

### 4.2.2 译码/派发阶段（Decode/Dispatch）

**功能**：
1. 译码指令，识别操作类型和操作数
2. 检查依赖关系（记分牌）
3. 决定哪些指令可以派发
4. 将指令发送到执行单元

**派发宽度**：每周期最多派发 4 条指令

**派发规则**：
- 必须连续派发（不能跳过）
- 检查数据依赖（RAW、WAW 冒险）
- 检查结构冒险（执行单元是否可用）
- 检查控制流依赖（跳转后的指令不派发）

**ASCII 图示**：

```
派发逻辑：

指令缓冲区                     执行单元
┌─────────┐                   ┌─────────┐
│ 指令 0  │ ──检查依赖──> 派发? ──> │ ALU(0)  │
└─────────┘       │                 └─────────┘
                  v
┌─────────┐    成功?               ┌─────────┐
│ 指令 1  │ ──检查依赖──> 派发? ──> │ ALU(1)  │
└─────────┘       │                 └─────────┘
                  v
┌─────────┐    成功?               ┌─────────┐
│ 指令 2  │ ──检查依赖──> 派发? ──> │ MLU     │
└─────────┘       │                 └─────────┘
                  v
┌─────────┐    失败!               
│ 指令 3  │ ──不尝试──> 暂停
└─────────┘

lastReady 链保证连续性：
  lastReady(0) = true
  lastReady(1) = 指令0派发成功
  lastReady(2) = 指令1派发成功
  lastReady(3) = 指令2派发成功
```

### 4.2.3 执行/写回阶段（Execute/Writeback）

**功能**：
1. 从寄存器文件读取操作数
2. 执行计算
3. 将结果写回寄存器文件

**执行单元配置**：

| 执行单元 | 数量 | 延迟 | 流水线 | 吞吐量 | 说明 |
|---------|------|------|--------|--------|------|
| **ALU** | 4 | 1周期 | 否 | 4条/周期 | 算术逻辑运算 |
| **BRU** | 4 | 1周期 | 否 | 4条/周期 | 分支跳转 |
| **MLU** | 1 | 2周期 | 是 | 1条/周期 | 乘法器（流水线）|
| **DVU** | 1 | 变化 | 否 | 1条/变化周期 | 除法器（非流水线）|
| **LSU** | 1 | 2+ | 队列 | 多条排队 | 加载存储单元 |

**注意**：
- **流水线执行单元**（如 MLU）：可以每周期接受新指令，但结果需要多周期后才能得到
- **非流水线执行单元**（如 DVU）：必须等待当前指令完成才能接受新指令

## 4.3 分发机制（Dispatch）

### 4.3.1 顺序多发原理

**核心思想**：按照程序顺序尝试派发指令，直到遇到无法派发的指令为止。

**lastReady 链机制**：

```scala
// 伪代码
val lastReady = Wire(Vec(5, Bool()))
lastReady(0) := true.B  // 第一个槽总是可以尝试

for (i <- 0 until 4) {
  val tryDispatch = lastReady(i) && canDispatch(i)
  
  // 尝试派发到执行单元
  val dispatched = 派发成功?
  
  lastReady(i + 1) := dispatched  // 传递给下一个槽
}
```

**工作原理**：

```
示例：4 条指令

指令 0: ADD x1, x2, x3   (无依赖)
指令 1: DIV x4, x5, x6   (无依赖)
指令 2: MUL x7, x1, x8   (依赖 x1，指令 0 写)
指令 3: ADD x9, x10, x11 (无依赖)

派发过程：

lastReady(0) = true
  ↓
指令 0 尝试派发 -> 成功 ✓
  ↓
lastReady(1) = true
  ↓
指令 1 尝试派发 -> 成功 ✓
  ↓
lastReady(2) = true
  ↓
指令 2 尝试派发 -> 失败 ✗ (RAW 冒险，x1 忙)
  ↓
lastReady(3) = false
  ↓
指令 3 不尝试派发

结果：派发指令 0, 1（连续，到冒险为止）
```

### 4.3.2 记分牌机制（Scoreboard）

**什么是记分牌**：
记分牌是一个 32 位的位向量（bit vector），每一位对应一个寄存器。
- `scoreboard[i] = 1`：寄存器 xi 有待完成的写入（不能读）
- `scoreboard[i] = 0`：寄存器 xi 可以安全读取

**记分牌操作**：

```
设置（派发时）：
  当指令派发时，将其目标寄存器对应的位置 1
  scoreboard |= (1 << rd_addr)

清除（写回时）：
  当指令写回时，将其目标寄存器对应的位清 0
  scoreboard &= ~(1 << rd_addr)

检查（派发时）：
  检查源寄存器对应的位是否为 1
  if (scoreboard & (1 << rs1_addr)) != 0:
      -> RAW 冒险，暂停
```

**ASCII 图示**：

```
记分牌状态示例：

寄存器:  x0  x1  x2  x3  x4  x5  ...  x31
记分牌:  0   1   0   0   1   0  ...   0
         ↑   ↑           ↑
         |   |           |
      总是0  忙         忙

解释：
  - x0 总是 0（RISC-V 规定）
  - x1 有待写入（某条指令正在计算结果）
  - x4 有待写入
  - 其他寄存器可以安全读取
```

### 4.3.3 依赖检查

**三种数据冒险**：

1. **RAW（Read After Write）冒险**：
   - 后面的指令读取前面指令要写的寄存器
   - 例子：
     ```
     指令 1: ADD x1, x2, x3   (写 x1)
     指令 2: SUB x4, x1, x5   (读 x1)  <- RAW 冒险
     ```
   - 解决：检查记分牌，如果 x1 忙则暂停指令 2

2. **WAW（Write After Write）冒险**：
   - 两条指令写同一个寄存器
   - 例子：
     ```
     指令 1: ADD x1, x2, x3   (写 x1)
     指令 2: SUB x1, x4, x5   (写 x1)  <- WAW 冒险
     ```
   - 解决：检查记分牌，如果 x1 忙则暂停指令 2

3. **WAR（Write After Read）冒险**：
   - 后面的指令写前面指令要读的寄存器
   - 例子：
     ```
     指令 1: ADD x3, x1, x2   (读 x1)
     指令 2: SUB x1, x4, x5   (写 x1)  <- WAR 冒险
     ```
   - **CoralNPU 中不会发生**：因为所有指令在派发后的下一个周期就读取操作数

**为什么 CoralNPU 不会发生 WAR 冒险**：

```
时间线：

周期 1: 指令 1, 2 派发
周期 2: 指令 1 读 x1, x2（从寄存器文件）
        指令 2 读 x4, x5（从寄存器文件）
周期 3: 指令 1 写 x3
        指令 2 写 x1

关键：所有指令在同一周期读取操作数，
     写回在后面的周期，不会冲突
```


## 4.4 乱序执行（Out-of-Order Execution）

### 4.4.1 为什么需要乱序执行

**问题**：不同指令的执行延迟不同

```
示例程序：

指令 0: ADD x1, x2, x3   (ALU, 1 周期)
指令 1: DIV x4, x5, x6   (DVU, 32 周期)
指令 2: MUL x7, x8, x9   (MLU, 2 周期)
指令 3: ADD x10, x11, x12 (ALU, 1 周期)

如果顺序执行：
  周期 1: 指令 0 执行
  周期 2: 指令 0 完成，指令 1 开始
  周期 33: 指令 1 完成，指令 2 开始
  周期 35: 指令 2 完成，指令 3 开始
  周期 36: 指令 3 完成
  总时间：36 周期

如果乱序执行：
  周期 1: 指令 0, 1 派发
  周期 2: 指令 0 完成，指令 2, 3 派发
  周期 3: 指令 3 完成
  周期 4: 指令 2 完成
  周期 33: 指令 1 完成
  总时间：33 周期

加速：约 9%
```

**乱序执行的本质**：
- 让快的指令先完成，不等慢的指令
- 充分利用多个执行单元的并行性
- 隐藏长延迟操作（除法、访存）

### 4.4.2 乱序完成的原因

**原因 1：不同的执行延迟**

```
执行单元延迟对比：

ALU:  ████ 1 周期
BRU:  ████ 1 周期
MLU:  ████████ 2 周期
DVU:  ████████████████████████████████ 32 周期
LSU:  ████████ 2+ 周期（取决于内存）

同时派发的指令会在不同时间完成
```

**原因 2：并行执行**

```
多个执行单元同时工作：

时间 ->
ALU(0) ━━ 指令0 ━━ 完成
ALU(1) ━━━━━━━━━━━━━━  (空闲)
ALU(2) ━━━━━━━━━━━━━━  (空闲)
ALU(3) ━━ 指令3 ━━ 完成
MLU    ━━━━ 指令2 ━━━━ 完成
DVU    ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 指令1 ━━ 完成

完成顺序：0 -> 3 -> 2 -> 1
派发顺序：0 -> 1 -> 2 -> 3
```

**原因 3：依赖链的影响**

```
如果指令 2 依赖指令 1：

指令 1: DIV x4, x5, x6   (32 周期)
指令 2: ADD x7, x4, x8   (依赖 x4，必须等待)
指令 3: MUL x9, x10, x11 (无依赖，可以先执行)

执行时间线：
  指令 1: 周期 1-33（DVU 慢慢算）
  指令 2: 暂停等待（记分牌阻止）
  指令 3: 周期 1 派发，周期 2-4 执行，周期 4 完成

完成顺序：3 -> 1 -> 2
```

### 4.4.3 顺序分发、乱序完成、顺序提交

```
             │                         │

```

## 4.5 退休缓冲区（Retirement Buffer / ROB）

### 4.5.1 ROB 的作用

**退休缓冲区（Retirement Buffer，也叫 Reorder Buffer，ROB）** 是乱序执行处理器的核心组件。

**主要功能**：
1. **缓冲乱序完成的结果**：指令可能乱序完成，但结果先存在 ROB 中
2. **保证顺序提交**：只有当指令及其之前的所有指令都完成时，才能提交
3. **实现精确异常**：异常发生时，可以准确恢复到异常指令的状态
4. **支持推测执行**：分支预测错误时，可以取消后续指令

### 4.5.2 ROB 结构

CoralNPU 的 ROB 采用**双缓冲区设计**：

```
ROB 结构：

┌─────────────────────────────────────────────────────────┐
│                   指令信息缓冲区                         │
│                   (instBuffer)                          │
│  存储：PC、指令编码、目标寄存器索引、陷阱标志            │
│  类型：FIFO（先进先出）                                  │
│  顺序：按程序顺序排列                                    │
└─────────────────────────────────────────────────────────┘
                           │
                           │ 对应关系
                           │
┌─────────────────────────────────────────────────────────┐
│                   结果缓冲区                             │
│                   (resultBuffer)                        │
│  存储：执行结果、完成标志、陷阱标志                      │
│  类型：寄存器数组                                        │
│  更新：乱序更新（指令完成时）                            │
└─────────────────────────────────────────────────────────┘

对应关系：
  instBuffer[0] <-> resultBuffer[0]  (最老的指令，head)
  instBuffer[1] <-> resultBuffer[1]
  ...
  instBuffer[n-1] <-> resultBuffer[n-1]  (最新的指令，tail)
```

**数据结构**：

```scala
// 指令信息（instBuffer 中的每一项）
class Instruction {
  val addr: UInt(32.W)           // PC 地址
  val inst: UInt(32.W)           // 指令编码
  val idx: UInt(idxWidth.W)      // 目标寄存器索引
  val trap: Bool                 // 陷阱标志
  val isControlFlow: Bool        // 是否是控制流指令
  val isBranch: Bool             // 是否是分支指令
}

// 执行结果（resultBuffer 中的每一项）
class InstructionUpdate {
  val result: UInt(32.W)         // 执行结果
  val trap: Bool                 // 陷阱标志
  val cfDone: Bool               // 控制流检查完成
}
```

### 4.5.3 ROB 操作

**入队（Enqueue）**：

```
派发时：
  1. 将指令信息写入 instBuffer 的 tail
  2. 初始化 resultBuffer[tail] = {valid=false}
  3. tail 指针后移

示例：
  派发指令 0, 1

  instBuffer:
    [0]: {addr=0x1000, idx=1}  <- head
    [1]: {addr=0x1004, idx=4}  <- tail

  resultBuffer:
    [0]: {valid=false}
    [1]: {valid=false}
```

**更新（Update）**：

```
指令完成时：
  1. 找到对应的 ROB 项（通过 PC 或寄存器索引匹配）
  2. 更新 resultBuffer[i] = {valid=true, result=...}

示例：
  指令 0 完成（周期 2）

  resultBuffer:
    [0]: {valid=true, result=100}  <- 完成
    [1]: {valid=false}             <- 未完成
```

**出队（Dequeue）**：

```
退休时：
  1. 从 head 开始检查连续完成的指令
  2. 将结果写回寄存器文件
  3. head 指针后移

示例：
  退休指令 0

  写回：x1 = 100
  instBuffer 和 resultBuffer 同时出队
  head 指针后移
```

### 4.5.4 顺序退休算法

**核心算法：连续完成检测（Cto）**

```scala
// 计算从 head 开始连续完成的指令数
val countValid = Cto(VecInit(resultBuffer.map(_.valid)).asUInt)

// Cto: Count Trailing Ones（计算连续的 1）
// 从 LSB（head）开始计算连续完成的指令数
```

**示例**：


### 4.5.5 精确异常（Precise Exception）

**什么是精确异常**：
当异常发生时，处理器的状态满足：
1. 异常之前的所有指令都已完成并提交
2. 异常指令（可能）完成但不提交状态改变
3. 异常之后的所有指令被取消

**为什么需要精确异常**：
- **程序语义正确**：保证异常处理程序看到的状态是一致的
- **可调试性**：调试器可以准确定位异常位置
- **RISC-V 规范要求**：RISC-V 规定必须实现精确异常

**实现方式**：

```
精确异常实现流程：

1. 检测异常
   当指令执行时检测到异常（如除零、非法指令）
   标记 ROB 项的 trap 标志

2. 继续执行
   继续正常执行和退休其他指令
   异常指令在 ROB 中等待

3. 等待到 head
   当异常指令到达 ROB 的 head 位置时
   检查异常之前的所有指令是否都已完成

4. 触发异常
   退休异常之前的所有指令
   触发 trap_flush 信号
   清空 ROB 和所有流水线
   PC 跳转到异常处理程序（mtvec）

5. 保存状态
   mepc = 异常指令的 PC
   mcause = 异常类型
   mtval = 异常相关信息
```

**示例**：

```
程序：
  指令 0: ADD x1, x2, x3   (正常)
  指令 1: LOAD x4, 0(x5)   (正常)
  指令 2: DIV x6, x7, x0   (除零异常！)
  指令 3: SUB x8, x9, x10  (后续指令)
  指令 4: MUL x11, x12, x13 (后续指令)

执行时间线：

周期 1: 派发指令 0, 1
周期 2: 派发指令 2, 3
        指令 0 完成
周期 3: 派发指令 4
        指令 1 完成
        指令 2 检测到除零异常，标记 trap=true
周期 4: 指令 3 完成
        指令 4 执行中
周期 5: 退休指令 0（正常）
周期 6: 退休指令 1（正常）
周期 7: 指令 2 到达 head
        检测到 trap=true
        触发异常处理：
          - 保存 mepc = 指令 2 的 PC
          - 保存 mcause = 除零异常
          - 清空 ROB（取消指令 3, 4）
          - PC = mtvec（跳转异常处理）

结果：
  ✓ 指令 0, 1 已提交（异常前）
  ✓ 指令 2 触发异常（异常点）
  ✗ 指令 3, 4 被取消（异常后）

精确！
```

## 4.6 记分牌（Scoreboard）

### 4.6.1 记分牌原理

**记分牌**是一个简单但高效的依赖跟踪机制，用于检测数据冒险。

**数据结构**：
- 32 位位向量（bit vector）
- 每一位对应一个寄存器（x0-x31）
- `scoreboard[i] = 1` 表示寄存器 xi 有待完成的写入

**ASCII 图示**：

```
记分牌结构：

寄存器编号:  31  30  29  ...  5   4   3   2   1   0
记分牌位:    0   0   1   ...  0   1   0   0   1   0
                     ↑           ↑           ↑   ↑
                     |           |           |   |
                   x29 忙      x4 忙       x1 忙 x0 总是 0

解释：
  - x0 总是 0（RISC-V 规定，硬件强制）
  - x1, x4, x29 有待写入（某些指令正在计算结果）
  - 其他寄存器可以安全读取
```

### 4.6.2 记分牌操作

**设置（Set）**：

```scala
// 派发时，将目标寄存器对应的位置 1
val scoreboard_set = io.writeAddr
    .map(x => MuxOR(x.valid, UIntToOH(x.addr, 32)))
    .reduce(_|_)

// UIntToOH: 将地址转换为 one-hot 编码
// 例如：addr=5 -> 0b00000000000000000000000000100000
```

**清除（Clear）**：

```scala
// 写回时，将目标寄存器对应的位清 0
val scoreboard_clr = io.writeData
    .map(x => MuxOR(x.valid, UIntToOH(x.bits.addr, 32)))
    .reduce(_|_)
```

**更新**：

```scala
// 每周期更新记分牌
when (scoreboard_set =/= 0.U || scoreboard_clr =/= 0.U) {
  val nxtScoreboard = (scoreboard & ~scoreboard_clr) | scoreboard_set
  scoreboard := Cat(nxtScoreboard(31,1), 0.U(1.W))  // x0 强制为 0
}
```

**示例**：

```
初始状态：
  scoreboard = 0b00000000000000000000000000000000

周期 1：派发指令 ADD x1, x2, x3
  scoreboard_set = 0b00000000000000000000000000000010 (x1)
  scoreboard = 0b00000000000000000000000000000010

周期 2：派发指令 DIV x4, x5, x6
        指令 ADD 完成，写回 x1
  scoreboard_set = 0b00000000000000000000000000010000 (x4)
  scoreboard_clr = 0b00000000000000000000000000000010 (x1)
  scoreboard = (0b00000010 & ~0b00000010) | 0b00010000
             = 0b00000000 | 0b00010000
             = 0b00000000000000000000000000010000

周期 3：尝试派发指令 SUB x7, x4, x8
  检查 x4: scoreboard[4] = 1 (忙)
  -> RAW 冒险，暂停
```

## 4.7 冒险检测（Hazard Detection）

### 4.7.1 三种数据冒险详解

**1. RAW（Read After Write）冒险**

**定义**：后面的指令读取前面指令要写的寄存器。

**为什么叫 RAW**：
- Read After Write = 在写之后读
- 后面的指令需要读取前面指令写入的值
- 如果前面的指令还没写完，后面的指令就读，会读到旧值（错误）

**示例**：

```
指令 1: ADD x1, x2, x3   (写 x1)
指令 2: SUB x4, x1, x5   (读 x1)  <- RAW 冒险

时间线（无保护）：
  周期 1: 指令 1 派发，x1 标记为忙
  周期 2: 指令 2 派发，读 x1 -> 读到旧值（错误！）
  周期 3: 指令 1 写回 x1 -> 新值

时间线（有保护）：
  周期 1: 指令 1 派发，x1 标记为忙
  周期 2: 指令 2 尝试派发，检查 scoreboard[1]=1 -> 暂停
  周期 3: 指令 1 写回 x1，scoreboard[1]=0
  周期 4: 指令 2 重试派发，检查 scoreboard[1]=0 -> 成功
```

**2. WAW（Write After Write）冒险**

**定义**：两条指令写同一个寄存器。

**为什么叫 WAW**：
- Write After Write = 在写之后写
- 两条指令都要写同一个寄存器
- 如果乱序完成，可能导致写入顺序错误

**示例**：

```
指令 1: ADD x1, x2, x3   (写 x1，延迟 1 周期)
指令 2: DIV x1, x4, x5   (写 x1，延迟 32 周期)

时间线（无保护）：
  周期 1: 指令 1, 2 派发
  周期 2: 指令 1 完成，写 x1 = 100
  周期 33: 指令 2 完成，写 x1 = 25
  最终 x1 = 25（正确）

  但如果指令 2 先完成：
  周期 2: 指令 2 完成，写 x1 = 25
  周期 3: 指令 1 完成，写 x1 = 100
  最终 x1 = 100（错误！应该是 25）

时间线（有保护）：
  周期 1: 指令 1 派发，x1 标记为忙
  周期 2: 指令 2 尝试派发，检查 scoreboard[1]=1 -> 暂停
  周期 3: 指令 1 写回 x1，scoreboard[1]=0
  周期 4: 指令 2 重试派发，检查 scoreboard[1]=0 -> 成功
```

**3. WAR（Write After Read）冒险**

**定义**：后面的指令写前面指令要读的寄存器。

**为什么叫 WAR**：
- Write After Read = 在读之后写
- 后面的指令要写前面指令读取的寄存器
- 如果后面的指令先写，前面的指令就读到新值（错误）

**示例**：

```
指令 1: ADD x3, x1, x2   (读 x1)
指令 2: SUB x1, x4, x5   (写 x1)  <- WAR 冒险

时间线（可能出错）：
  周期 1: 指令 1, 2 派发
  周期 2: 指令 2 先写 x1 = 50
  周期 3: 指令 1 读 x1 = 50（错误！应该读旧值）

时间线（CoralNPU）：
  周期 1: 指令 1, 2 派发
  周期 2: 指令 1 读 x1 = 10（旧值）
          指令 2 读 x4, x5
  周期 3: 指令 1 写 x3
          指令 2 写 x1 = 50

关键：所有指令在派发后的下一个周期就读取操作数
     写回在后面的周期，不会冲突
```

**CoralNPU 为什么不会发生 WAR 冒险**：

```
CoralNPU 的读写时序：

周期 1: 派发
  - 指令进入执行单元
  - 记分牌标记目标寄存器

周期 2: 读操作数
  - 所有指令同时从寄存器文件读取操作数
  - 此时还没有任何写回

周期 3+: 写回
  - 指令完成后写回寄存器文件
  - 此时所有指令都已经读完操作数

结论：读和写在不同的时间窗口，不会冲突
```

### 4.7.2 结构冒险（Structural Hazard）

**定义**：多条指令竞争同一个硬件资源。

**CoralNPU 中的结构冒险**：

```
1. 执行单元数量限制
   - 只有 1 个 MLU，每周期最多派发 1 条乘法指令
   - 只有 1 个 DVU，必须等待当前除法完成

2. LSU 队列限制
   - LSU 有队列深度限制
   - 如果队列满，必须等待

3. ROB 空间限制
   - ROB 有固定大小（如 16 项）
   - 如果 ROB 满，必须等待退休

4. 寄存器文件端口限制
   - 寄存器文件有固定的读写端口数
   - 如果端口不够，必须限制派发数量
```

**解决方法**：

```
1. 增加硬件资源
   - 增加执行单元数量
   - 增加 ROB 大小
   - 增加寄存器文件端口

2. 限制派发
   - 检测资源是否可用
   - 如果不可用，暂停派发

3. 仲裁机制
   - 多个请求竞争同一资源时
   - 使用优先级或轮询选择
```


### 4.7.3 控制流冒险（Control Hazard）

**定义**：分支和跳转指令导致的流水线停顿。

**问题**：

```
程序：
  0x1000: BEQ x1, x2, target   (分支指令)
  0x1004: ADD x3, x4, x5       (后续指令)
  0x1008: SUB x6, x7, x8       (后续指令)
  ...
  0x2000: MUL x9, x10, x11     (target: 分支目标)

问题：
  - 周期 1: 取指令 0x1000 (BEQ)
  - 周期 2: 执行 BEQ，计算是否跳转
  - 周期 3: 知道结果，但已经取了 0x1004, 0x1008
  
  如果跳转：
    - 0x1004, 0x1008 是错误的指令
    - 必须取消，重新取 0x2000
    - 浪费 2 个周期
```

**解决方法**：

1. **分支预测**：猜测分支是否跳转，提前取指令
2. **延迟槽**：在分支后插入无关指令（RISC-V 不使用）
3. **停顿**：等待分支结果，再取下一条指令（简单但慢）

## 4.8 数据前递（Forwarding）

### 4.8.1 什么是数据前递

**数据前递（Forwarding，也叫 Bypassing）** 是一种优化技术，用于减少 RAW 冒险导致的停顿。

**核心思想**：
- 不等指令写回寄存器文件
- 直接将结果转发给需要的指令
- 消除等待延迟

**示例**：

```
无前递（需要暂停）：

指令 1: ADD x1, x2, x3   (写 x1)
指令 2: SUB x4, x1, x5   (读 x1)

周期 1: 指令 1 派发
周期 2: 指令 1 执行，计算 x1 = 100
        指令 2 尝试派发，检查 scoreboard[1]=1 -> 暂停
周期 3: 指令 1 写回 x1 = 100，scoreboard[1]=0
周期 4: 指令 2 重试派发，读 x1 = 100 -> 成功
周期 5: 指令 2 执行

延迟：3 周期（从指令 1 派发到指令 2 派发）

有前递（零延迟）：

周期 1: 指令 1 派发
周期 2: 指令 1 执行，计算 x1 = 100
        指令 2 派发，直接获取指令 1 的结果（前递）
周期 3: 指令 2 执行，使用 x1 = 100

延迟：1 周期（从指令 1 派发到指令 2 派发）

性能提升：消除 2 周期停顿
```

### 4.8.2 CoralNPU 的前递机制

**实现位置**：寄存器文件（Regfile.scala）

```scala
// 检查寄存器是否正在被写
val write = VecAt(writeValid, idx)

// 选择数据源
rwdata(i) := Mux(write, wdata(i), rdata(i))

// rwdata = 带前递的数据
//   如果寄存器正在被写入：使用新值（wdata，前递）
//   否则：使用寄存器旧值（rdata）
```

**前递路径图**：

```
前递路径：

执行单元                    寄存器文件
┌─────────┐                ┌─────────┐
│  ALU    │ ─写回端口─>    │ 写端口  │
│ 指令 1  │                │ (wdata) │
└─────────┘                └────┬────┘
                                │
                                │ 前递路径
                                │
                                v
                           ┌─────────┐
                           │  MUX    │
                           │ write?  │
                           └────┬────┘
                                │
                    ┌───────────┴───────────┐
                    │                       │
                    v                       v
              wdata (新值)            rdata (旧值)
                    │                       │
                    └───────────┬───────────┘
                                │
                                v
                           ┌─────────┐
                           │ rwdata  │
                           │ (输出)  │
                           └────┬────┘
                                │
                                v
                           ┌─────────┐
                           │  ALU    │
                           │ 指令 2  │
                           └─────────┘
```

**前递条件**：

```
前递发生的条件：

1. 写端口有效（writeValid = true）
2. 写地址匹配读地址（writeAddr == readAddr）
3. 在同一周期

检查逻辑：
  for each 读端口 i:
    for each 写端口 j:
      if (writeValid[j] && writeAddr[j] == readAddr[i]):
        rwdata[i] = wdata[j]  // 前递
        break
    if (没有匹配的写端口):
      rwdata[i] = rdata[i]  // 正常读取
```

**前递的限制**：

```
CoralNPU 的前递只在同一周期内有效：

周期 1: 指令 1 派发
周期 2: 指令 1 写回，指令 2 派发
        -> 可以前递（同一周期）

周期 1: 指令 1 派发
周期 2: 指令 1 执行中
周期 3: 指令 1 写回，指令 2 派发
        -> 不能前递（不同周期）
        -> 必须通过记分牌暂停
```

### 4.8.3 前递的性能影响

**性能提升**：

```
典型程序中的 RAW 冒险比例：
  - 约 20-30% 的指令有 RAW 依赖
  - 无前递：每个 RAW 冒险导致 1-2 周期停顿
  - 有前递：大部分 RAW 冒险零延迟

性能提升估算：
  - 无前递 IPC：约 2.0
  - 有前递 IPC：约 2.5-2.8
  - 提升：25-40%
```

**硬件成本**：

```
前递逻辑的硬件成本：

1. 比较器
   - 每个读端口需要与所有写端口比较
   - 读端口数 × 写端口数 × 5 位比较器
   - 例如：8 读 × 6 写 × 5 位 = 240 位比较

2. 多路选择器
   - 每个读端口需要一个 MUX
   - 选择写端口数据或寄存器数据
   - 例如：8 个 (6+1)-to-1 MUX × 32 位

3. 总成本
   - 约 500-1000 gates
   - 占寄存器文件面积的 10-15%

投资回报率：极高！
```

## 4.9 分支预测（Branch Prediction）

### 4.9.1 为什么需要分支预测

**问题**：分支指令导致流水线停顿

```
程序：
  0x1000: BEQ x1, x2, 0x2000   (分支指令)
  0x1004: ADD x3, x4, x5       (不跳转路径)
  ...
  0x2000: MUL x6, x7, x8       (跳转路径)

无预测：
  周期 1: 取 0x1000 (BEQ)
  周期 2: 执行 BEQ，等待结果
  周期 3: 知道结果，取下一条指令
  
  停顿：2 周期

有预测：
  周期 1: 取 0x1000 (BEQ)，预测跳转，取 0x2000
  周期 2: 执行 BEQ，验证预测
  周期 3: 如果预测正确，继续执行 0x2000
          如果预测错误，取消 0x2000，取 0x1004
  
  预测正确：0 周期停顿
  预测错误：2 周期停顿
```

### 4.9.2 CoralNPU 的静态分支预测

**策略**：基于分支方向的静态预测

```
静态预测规则：

1. 后向分支（地址减小）：预测为跳转
   - 原因：循环通常会执行多次
   - 例子：for (i=0; i<100; i++) { ... }
           循环体结束时跳回开头（后向分支）
           99 次跳转，1 次不跳转
           预测跳转的准确率：99%

2. 前向分支（地址增大）：预测为不跳转
   - 原因：if 语句通常不执行
   - 例子：if (error) { handle_error(); }
           大部分情况下 error=false
           预测不跳转的准确率：70-80%
```

**实现**：

```scala
// Fetch.scala
val brchFwd = Wire(Vec(4, Bool()))

for (i <- 0 until 4) {
  // 计算分支目标地址
  val target = instAddr(i) + offset
  
  // 判断分支方向
  val backward = target < instAddr(i)
  
  // 预测：后向跳转，前向不跳转
  brchFwd(i) := backward
}
```


## 4.10 性能计数器（Performance Counters）

### 4.10.1 什么是性能计数器

**性能计数器**是一组硬件寄存器，用于统计处理器的运行状态和性能指标。

**用途**：
- **性能分析**：找出程序的性能瓶颈
- **调试**：验证硬件是否正常工作
- **优化**：评估优化效果

### 4.10.2 CoralNPU 的性能计数器

**RISC-V 标准计数器**：

| 名称 | 地址 | 说明 |
|------|------|------|
| `mcycle` | 0xB00 | 周期计数器（低 32 位） |
| `mcycleh` | 0xB80 | 周期计数器（高 32 位） |
| `minstret` | 0xB02 | 已退休指令计数器（低 32 位） |
| `minstreth` | 0xB82 | 已退休指令计数器（高 32 位） |

**自定义计数器**（如果实现）：

```
可能的自定义计数器：

1. 派发计数器
   - 统计每周期派发的指令数
   - 用于评估派发效率

2. 停顿计数器
   - 统计因各种冒险导致的停顿周期
   - 分类：RAW 停顿、WAW 停顿、结构停顿

3. 分支预测计数器
   - 统计分支预测的准确率
   - 分支总数、预测正确数、预测错误数

4. 缓存计数器
   - 统计缓存命中率
   - L0 命中、L0 缺失

5. ROB 计数器
   - 统计 ROB 的使用情况
   - 平均占用率、最大占用率
```

### 4.10.3 使用性能计数器

**读取计数器**：

```c
// 读取周期计数器
uint64_t read_mcycle() {
    uint32_t lo, hi;
    asm volatile("csrr %0, mcycle" : "=r"(lo));
    asm volatile("csrr %0, mcycleh" : "=r"(hi));
    return ((uint64_t)hi << 32) | lo;
}

// 读取指令计数器
uint64_t read_minstret() {
    uint32_t lo, hi;
    asm volatile("csrr %0, minstret" : "=r"(lo));
    asm volatile("csrr %0, minstreth" : "=r"(hi));
    return ((uint64_t)hi << 32) | lo;
}
```

**计算 IPC**：

```c
// 计算 IPC (Instructions Per Cycle)
double calculate_ipc() {
    uint64_t start_cycle = read_mcycle();
    uint64_t start_instret = read_minstret();
    
    // 运行程序
    run_program();
    
    uint64_t end_cycle = read_mcycle();
    uint64_t end_instret = read_minstret();
    
    uint64_t cycles = end_cycle - start_cycle;
    uint64_t instructions = end_instret - start_instret;
    
    return (double)instructions / (double)cycles;
}
```

**性能分析示例**：

```c
// 性能分析
void performance_analysis() {
    uint64_t cycles = read_mcycle();
    uint64_t instructions = read_minstret();
    
    printf("Cycles: %llu\n", cycles);
    printf("Instructions: %llu\n", instructions);
    printf("IPC: %.2f\n", (double)instructions / cycles);
    
    // 如果 IPC < 1.0：可能有大量停顿
    // 如果 IPC > 2.0：派发效率高
    // 如果 IPC > 3.0：接近理论上限
}
```

## 4.12 微架构设计权衡

### 4.12.1 性能 vs 成本

**流水线级数**：

```
浅流水线（3 级）：
  优点：
    - 控制逻辑简单
    - 分支预测错误代价小

  优点：

  缺点：
    - 控制逻辑复杂

    - 功耗高
```

**派发宽度**：

```

  优点：硬件简单
  缺点：性能低

双发射（2 条/周期）：
  优点：性能提升明显
  缺点：硬件成本增加 50%

四发射（4 条/周期）：
  优点：性能接近上限
```

```
  优点：平衡性能和成本
  缺点：-

```

### 4.12.2 顺序 vs 乱序

**顺序派发 vs 乱序派发**：

```
顺序派发：
  优点：
    - 控制逻辑简单
    - 易于验证
    - 异常处理简单
  缺点：
    - 可能因依赖暂停
    - 不能完全利用并行度

乱序派发：
  优点：
    - 可以跳过有依赖的指令
    - 充分利用并行度
  缺点：
    - 需要寄存器重命名
    - 控制逻辑复杂
    - 硬件成本高

CoralNPU 选择：顺序派发（简单可靠）
```

**顺序提交 vs 乱序提交**：

```
顺序提交：
  优点：
    - 精确异常
    - 程序语义正确
    - 易于调试
  缺点：
    - 需要 ROB
    - 可能因前面的指令阻塞

乱序提交：
  优点：
    - 不需要 ROB
    - 性能略高
  缺点：
    - 不支持精确异常
    - 程序语义可能错误
    - 难以调试

CoralNPU 选择：顺序提交（正确性优先）
```

## 4.13 总结

### 4.13.1 微架构核心特性

CoralNPU 采用**顺序多发、乱序完成、顺序提交**架构：

```
✅ 顺序多发：
  - lastReady 链保证连续性
  - 记分牌快速依赖检查
  - 数据前递消除等待

✅ 乱序完成：
  - 多个执行单元并行
  - 不同延迟独立完成
  - ROB 缓冲结果

✅ 顺序提交：
  - Cto 算法检测连续完成
  - 精确异常处理
  - 按序写回寄存器文件
```

### 4.13.2 关键技术

```
1. 记分牌（Scoreboard）
   - 32 位位向量
   - 快速依赖检查
   - 低硬件成本

2. 退休缓冲区（ROB）
   - 双缓冲区设计
   - 缓冲乱序结果
   - 保证顺序提交

3. 数据前递（Forwarding）
   - 同周期转发
   - 消除 RAW 停顿
   - 性能提升 25-40%

4. 分支预测（Branch Prediction）
   - 静态预测策略
```

### 4.13.3 性能特点

```
理论性能：
  - 派发宽度：4 条/周期
  - 理论 IPC：4.0

实际性能：
  - 典型 IPC：2.0-2.8
  - 效率：50-70%

性能损失原因：
  - 数据依赖：30-40%
  - 控制流依赖：15-20%
  - 结构冒险：10-15%
  - 内存延迟：5-10%
```

### 4.13.4 设计哲学

CoralNPU 的微架构设计遵循以下原则：

```
1. 简单性优先
   - 顺序派发简化控制逻辑
   - 易于验证和调试

2. 正确性保证
   - 顺序提交保证程序语义
   - 精确异常支持调试

3. 性能平衡
   - 乱序完成提升性能
   - 多发射增加吞吐量

4. 成本可控
   - 合理的 ROB 大小
   - 高效的记分牌机制
```

这种架构在**性能、成本、正确性**之间取得了最佳平衡，是现代高性能处理器的标准设计模式。

---

**本章要点**：

1. **微架构**是处理器的具体实现方式，与**架构**（指令集规范）不同
2. CoralNPU 采用**三级流水线**：取指、译码/派发、执行/写回
3. **顺序多发**通过 lastReady 链保证连续派发
4. **记分牌**用于快速检测 RAW 和 WAW 冒险
5. **乱序完成**允许短延迟指令先完成，隐藏长延迟
6. **ROB**缓冲乱序结果，保证顺序提交和精确异常
7. **数据前递**消除 RAW 冒险的停顿，提升性能 25-40%
8. **静态分支预测**基于分支方向，准确率 70-80%
9. **性能计数器**用于监控和分析处理器性能
