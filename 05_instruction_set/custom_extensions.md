# 5.4 自定义扩展

## 5.4.1 自定义扩展概述

### 为什么需要自定义扩展

CoralNPU 是一个面向机器学习加速的 RISC-V 处理器。虽然标准的 RISC-V 指令集（包括 RV32IM 和 RVV）已经提供了丰富的计算能力，但机器学习工作负载有一些特殊的需求：

1. **量化计算**：神经网络推理通常使用 INT8 或更低精度的量化数据，需要专门的量化和反量化指令
2. **高效的 MAC 运算**：卷积和矩阵乘法是 ML 的核心操作，需要高效的乘累加（MAC）指令
3. **数据重排**：ML 模型经常需要对数据进行转置、打包、解包等操作
4. **批处理优化**：需要支持多批次数据的并行处理

为了满足这些需求，CoralNPU 设计了自定义扩展指令集。

### C 扩展编码空间的回收利用

RISC-V 规范允许不实现 C 扩展（压缩指令）的处理器回收其编码空间用于其他用途。CoralNPU 采用了这一策略：

- **标准 C 扩展编码空间**：`00`、`01`、`10` 开头的 16 位指令
- **CoralNPU 的选择**：不实现 C 扩展，将这部分编码空间用于 SIMD/向量指令

这样做的好处：
- 为 SIMD 寄存器提供 6 位索引空间（支持 64 个向量寄存器 v0-v63）
- 允许更灵活的指令编码格式
- 支持指令压缩和 stripmining 机制

> **注意**：这里说的"压缩"不是指 C 扩展的压缩指令，而是指将标准 RVV 指令的编码进行重新编码，以适应 CoralNPU 的内部流水线。

## 5.4.2 当前实现状态

### 标准 RVV 指令集

当前版本的 CoralNPU 主要使用标准的 RISC-V Vector (RVV) 1.0 指令集，包括：

- **向量算术指令**：VADD、VSUB、VMUL 等
- **向量逻辑指令**：VAND、VOR、VXOR 等
- **向量移位指令**：VSLL、VSRL、VSRA 等
- **向量比较指令**：VMSEQ、VMSLT、VMSLTU 等
- **向量饱和运算**：VSADD、VSSUB、VSMUL 等
- **向量缩减指令**：VNSRL、VNSRA、VNCLIP 等

这些指令已经能够满足大部分机器学习工作负载的需求。

### 内部指令压缩格式

CoralNPU 在内部使用了一种压缩的指令格式，将标准的 32 位 RVV 指令重新编码为更紧凑的格式：

```
标准 RVV 指令格式（32 位）：
31        25 24   20 19   15 14  12 11    7 6      0
+-----------+-------+-------+------+-------+--------+
|  funct6   |  vs2  |  vs1  |funct3|  vd   | opcode |
+-----------+-------+-------+------+-------+--------+

CoralNPU 内部压缩格式（27 位）：
26      25 24                                      0
+--------+------------------------------------------+
| opcode |              bits (25 位)                |
+--------+------------------------------------------+

opcode 编码：
  00 - RVVLOAD  (向量加载)
  01 - RVVSTORE (向量存储)
  10 - RVVALU   (向量运算)
```

这种压缩格式的优点：
- 节省指令缓存空间
- 简化译码逻辑
- 为未来的自定义指令预留空间


## 5.4.3 ML 专用指令（规划中）

虽然当前版本主要使用标准 RVV 指令，但 CoralNPU 的架构设计已经为未来的 ML 专用指令做好了准备。以下是规划中的自定义指令类型。

### 量化相关指令

量化是将浮点数转换为低精度整数的过程，反量化则是相反的操作。这些操作在神经网络推理中非常常见。

**规划中的量化指令**：

1. **VQUANT8** - 将 32 位数据量化为 8 位
   - 功能：`vd[i] = saturate((vs1[i] * scale) >> shift, -128, 127)`
   - 用途：将激活值或权重量化为 INT8

2. **VDEQUANT8** - 将 8 位数据反量化为 32 位
   - 功能：`vd[i] = (vs1[i] * scale) >> shift`
   - 用途：将量化后的数据恢复为更高精度

3. **VQUANT4** - 将数据量化为 4 位
   - 功能：支持 4 位量化，进一步减少内存占用
   - 用途：超低精度推理

**量化指令的编码格式**（示例）：

```
31        25 24   20 19   15 14  12 11    7 6      0
+-----------+-------+-------+------+-------+--------+
|  funct6   | scale | shift |funct3|  vd   | opcode |
+-----------+-------+-------+------+-------+--------+
  自定义     量化参数  移位量   011    目标   1010111
```

### 特殊的 MAC 指令

CoralNPU 的核心是一个量化的外积乘累加引擎。外积（outer product）结构可以在两个维度上进行广播，最大化计算与内存访问的比率。

**规划中的 MAC 指令**：

1. **VDOT8** - 8 位点积运算
   - 功能：`acc += vs1[0:3] · vs2[0:3]`（4 个 8 位乘法累加到 32 位）
   - 用途：高效的卷积和矩阵乘法

2. **VOUTER8** - 8 位外积运算
   - 功能：`acc[i][j] += vs1[i] * vs2[j]`（外积累加）
   - 用途：批量卷积，利用数据重用

3. **VMAC8** - 8 位乘累加
   - 功能：`vd[i] += vs1[i] * vs2[i]`
   - 用途：通用的 MAC 操作

**外积 MAC 的概念图**：

```
        vs2 (权重向量，"宽"维度)
         [w0] [w1] [w2] [w3]
           |    |    |    |
vs1[x0] --[*]--[*]--[*]--[*]-- acc[0][0:3]
vs1[x1] --[*]--[*]--[*]--[*]-- acc[1][0:3]
vs1[x2] --[*]--[*]--[*]--[*]-- acc[2][0:3]
vs1[x3] --[*]--[*]--[*]--[*]-- acc[3][0:3]
  |
"窄"维度
(批次)

每个 [*] 是一个 8 位乘法器
结果累加到 32 位累加器 acc[i][j]
```

这种结构的优势：
- **数据重用**：权重向量在"宽"维度广播，输入在"窄"维度广播
- **高吞吐量**：单周期执行 16 个 8 位乘法（4x4 外积）
- **低内存带宽**：每次只需读取 8 个 8 位数据，产生 16 个乘法结果

### 数据重排指令

机器学习模型经常需要对数据进行重新排列，例如：
- 将 NCHW 格式转换为 NHWC 格式
- 将数据打包为特定的 tile 格式
- 执行转置操作

**规划中的数据重排指令**：

1. **VPACK8** - 将多个 8 位数据打包
   - 功能：将 4 个 8 位数据打包到一个 32 位字中
   - 用途：数据格式转换

2. **VUNPACK8** - 解包 8 位数据
   - 功能：将 32 位字解包为 4 个 8 位数据
   - 用途：数据格式转换

3. **VTRANSPOSE** - 矩阵转置
   - 功能：对小矩阵（如 4x4）进行转置
   - 用途：数据布局转换

**数据打包示例**：

```
打包前（4 个 32 位寄存器，每个只用低 8 位）：
v0: [00 00 00 a0]
v1: [00 00 00 a1]
v2: [00 00 00 a2]
v3: [00 00 00 a3]

执行 VPACK8 v4, v0, v1, v2, v3

打包后（1 个 32 位寄存器）：
v4: [a3 a2 a1 a0]
```


## 5.4.4 编码空间分配

### 如何利用 C 扩展的编码空间

RISC-V 的 C 扩展使用 16 位编码，其 opcode 占用了以下编码空间：

```
标准 RISC-V 指令编码空间（低 2 位）：
00 - C 扩展（压缩指令）
01 - C 扩展（压缩指令）
10 - C 扩展（压缩指令）
11 - 标准 32 位指令
```

CoralNPU 不实现 C 扩展，因此可以将 `00`、`01`、`10` 编码空间用于其他用途。但实际上，CoralNPU 仍然使用标准的 32 位指令格式（低 2 位为 `11`），只是在内部进行了重新编码。

### 自定义指令的编码格式

CoralNPU 的自定义指令（如果实现）将遵循 RISC-V 的自定义扩展规范，使用预留的 opcode 空间。

**RISC-V 自定义扩展 opcode**：

```
opcode[6:0] = 0001011 (custom-0)
opcode[6:0] = 0101011 (custom-1)
opcode[6:0] = 1011011 (custom-2)
opcode[6:0] = 1111011 (custom-3)
```

**自定义指令编码格式示例**：

```
R 型格式（用于 VDOT8、VMAC8 等）：
31        25 24   20 19   15 14  12 11    7 6      0
+-----------+-------+-------+------+-------+--------+
|  funct7   |  vs2  |  vs1  |funct3|  vd   |0001011 |
+-----------+-------+-------+------+-------+--------+
  操作码      源寄存器2 源寄存器1 功能码  目标寄存器 custom-0

I 型格式（用于 VQUANT8、VDEQUANT8 等）：
31                  20 19   15 14  12 11    7 6      0
+----------------------+-------+------+-------+--------+
|      imm[11:0]       |  vs1  |funct3|  vd   |0001011 |
+----------------------+-------+------+-------+--------+
  立即数（量化参数）      源寄存器  功能码  目标寄存器 custom-0
```

### 编码空间分配表

以下是规划中的自定义指令编码分配：

| funct7    | funct3 | 指令名称    | 功能描述                |
|-----------|--------|-------------|-------------------------|
| 0000000   | 000    | VQUANT8     | 量化为 8 位             |
| 0000001   | 000    | VDEQUANT8   | 从 8 位反量化           |
| 0000010   | 000    | VQUANT4     | 量化为 4 位             |
| 0000011   | 000    | VDEQUANT4   | 从 4 位反量化           |
| 0000100   | 000    | VDOT8       | 8 位点积                |
| 0000101   | 000    | VOUTER8     | 8 位外积                |
| 0000110   | 000    | VMAC8       | 8 位乘累加              |
| 0000111   | 000    | VPACK8      | 打包 8 位数据           |
| 0001000   | 000    | VUNPACK8    | 解包 8 位数据           |
| 0001001   | 000    | VTRANSPOSE  | 矩阵转置                |

> **注意**：以上编码分配仅为示例，实际实现时可能会有所不同。

### Stripmining 机制

CoralNPU 使用了一种称为 "stripmining" 的机制来减少指令分发压力。Stripmining 是指将数组并行性折叠到可用的硬件并行性中。

**Stripmining 的工作原理**：

```
前端分发：vadd v0
         ↓
内部展开：vadd v0
         vadd v1
         vadd v2
         vadd v3
```

一条前端指令在内部被展开为 4 条连续的指令，分别操作 v0、v1、v2、v3。这样可以：
- 减少指令分发带宽需求（4 倍）
- 原生支持指令级的 tiling 模式
- 简化软件编程模型

**Stripmining 的编码**：

Stripmining 信息可以编码在指令的保留位中，或者通过特殊的配置寄存器来控制。例如：

```
31        25 24   20 19   15 14  12 11    7 6      0
+-----------+-------+-------+------+-------+--------+
|  funct6 |S|  vs2  |  vs1  |funct3|  vd   |1010111 |
+-----------+-------+-------+------+-------+--------+
            ↑
            Stripmine 位：
            0 = 不展开
            1 = 展开为 4 条指令
```


## 5.4.5 使用示例

### 如何在 C 代码中使用自定义指令

虽然自定义指令还在规划中，但我们可以展示如何在 C 代码中使用它们。RISC-V 提供了内联汇编（inline assembly）机制，允许在 C 代码中嵌入汇编指令。

### 内联汇编基础

内联汇编的基本语法：

```c
asm volatile (
    "汇编指令"
    : 输出操作数列表
    : 输入操作数列表
    : 被修改的寄存器列表
);
```

**语法说明**：
- `asm` 或 `__asm__`：内联汇编关键字
- `volatile`：告诉编译器不要优化这段代码
- 输出操作数：`"=r"(变量)` 表示将结果写入变量
- 输入操作数：`"r"(变量)` 表示从变量读取输入
- 被修改的寄存器：列出会被修改的寄存器

### 示例 1：使用量化指令

假设我们要将一个浮点数组量化为 INT8：

```c
#include <stdint.h>

// 量化函数：将 float 转换为 int8
void quantize_array(float *input, int8_t *output, int length, 
                    float scale, int shift) {
    for (int i = 0; i < length; i += 4) {
        // 先将 float 转换为 int32
        int32_t tmp[4];
        for (int j = 0; j < 4; j++) {
            tmp[j] = (int32_t)(input[i + j] * scale);
        }
        
        // 使用自定义的 VQUANT8 指令
        // 注意：这是示例代码，实际指令可能不同
        asm volatile (
            "vle32.v v0, (%0)\n"      // 加载 4 个 int32 到 v0
            "vquant8 v1, v0, %1, %2\n" // 量化：v1 = quantize(v0, scale, shift)
            "vse8.v v1, (%3)\n"        // 存储 4 个 int8
            :                          // 无输出操作数
            : "r"(tmp), "r"(scale), "r"(shift), "r"(&output[i])
            : "v0", "v1", "memory"     // 修改了 v0, v1 和内存
        );
    }
}
```

**代码解释**：
- `vle32.v v0, (%0)`：这是标准 RVV 指令，加载 32 位数据到向量寄存器 v0
  - `vle32.v` = Vector Load Element, 32-bit
  - `(%0)` 表示使用第 0 个输入操作数（tmp 的地址）
- `vquant8 v1, v0, %1, %2`：自定义量化指令（示例）
  - 将 v0 中的数据量化，结果存入 v1
  - `%1` 和 `%2` 是 scale 和 shift 参数
- `vse8.v v1, (%3)`：标准 RVV 指令，存储 8 位数据
  - `vse8.v` = Vector Store Element, 8-bit

### 示例 2：使用 MAC 指令进行卷积

使用外积 MAC 指令实现高效的卷积：

```c
// 简化的卷积函数（1D 卷积示例）
void conv1d_int8(int8_t *input, int8_t *weight, int32_t *output,
                 int input_len, int weight_len) {
    // 假设使用 4x4 外积 MAC
    for (int i = 0; i < input_len - weight_len + 1; i += 4) {
        for (int j = 0; j < weight_len; j += 4) {
            // 加载输入和权重
            asm volatile (
                "vle8.v v0, (%0)\n"        // 加载 4 个输入
                "vle8.v v1, (%1)\n"        // 加载 4 个权重
                "vouter8 acc0, v0, v1\n"   // 外积 MAC（示例指令）
                :
                : "r"(&input[i + j]), "r"(&weight[j])
                : "v0", "v1", "acc0"
            );
        }
        
        // 读取累加器结果
        asm volatile (
            "vmv.v.x v2, acc0\n"           // 将累加器移动到向量寄存器
            "vse32.v v2, (%0)\n"           // 存储结果
            :
            : "r"(&output[i])
            : "v2", "memory"
        );
    }
}
```

**代码解释**：
- `vouter8 acc0, v0, v1`：外积 MAC 指令（示例）
  - 计算 v0 和 v1 的外积，累加到 acc0
  - acc0 是一个特殊的累加器寄存器（8x8 个 32 位累加器）
- 外积操作一次计算多个输出，提高了计算效率

### 示例 3：使用数据重排指令

将数据从 NCHW 格式转换为 NHWC 格式：

```c
// 数据格式转换（简化版）
void nchw_to_nhwc(int8_t *nchw, int8_t *nhwc, int n, int c, int h, int w) {
    // 假设 c = 4（4 个通道）
    for (int i = 0; i < n * h * w; i++) {
        // NCHW: [n][c][h][w]
        // NHWC: [n][h][w][c]
        
        int8_t ch0 = nchw[i];
        int8_t ch1 = nchw[i + h * w];
        int8_t ch2 = nchw[i + 2 * h * w];
        int8_t ch3 = nchw[i + 3 * h * w];
        
        // 使用打包指令
        asm volatile (
            "vpack8 v0, %0, %1, %2, %3\n"  // 打包 4 个通道（示例指令）
            "vse8.v v0, (%4)\n"            // 存储打包后的数据
            :
            : "r"(ch0), "r"(ch1), "r"(ch2), "r"(ch3), "r"(&nhwc[i * 4])
            : "v0", "memory"
        );
    }
}
```

### 示例 4：完整的神经网络层

结合多个自定义指令实现一个完整的神经网络层：

```c
// 量化卷积层（INT8）
void quantized_conv_layer(
    int8_t *input,      // 输入特征图
    int8_t *weight,     // 卷积核权重
    int32_t *bias,      // 偏置
    int8_t *output,     // 输出特征图
    int in_h, int in_w, int in_c,   // 输入尺寸
    int out_c,          // 输出通道数
    int k_h, int k_w,   // 卷积核尺寸
    float scale,        // 量化参数
    int shift
) {
    int out_h = in_h - k_h + 1;
    int out_w = in_w - k_w + 1;
    
    // 遍历输出特征图
    for (int oc = 0; oc < out_c; oc++) {
        for (int oh = 0; oh < out_h; oh++) {
            for (int ow = 0; ow < out_w; ow++) {
                int32_t acc = bias[oc];
                
                // 遍历卷积核
                for (int kh = 0; kh < k_h; kh++) {
                    for (int kw = 0; kw < k_w; kw++) {
                        for (int ic = 0; ic < in_c; ic += 4) {
                            // 使用 VDOT8 指令加速
                            asm volatile (
                                "vle8.v v0, (%0)\n"      // 加载输入
                                "vle8.v v1, (%1)\n"      // 加载权重
                                "vdot8 %2, v0, v1\n"     // 点积累加（示例）
                                : "+r"(acc)
                                : "r"(&input[(oh+kh)*in_w*in_c + (ow+kw)*in_c + ic]),
                                  "r"(&weight[oc*k_h*k_w*in_c + kh*k_w*in_c + kw*in_c + ic])
                                : "v0", "v1"
                            );
                        }
                    }
                }
                
                // 量化输出
                asm volatile (
                    "vquant8 v2, %0, %1, %2\n"   // 量化（示例）
                    "vse8.v v2, (%3)\n"          // 存储
                    :
                    : "r"(acc), "r"(scale), "r"(shift),
                      "r"(&output[oc*out_h*out_w + oh*out_w + ow])
                    : "v2", "memory"
                );
            }
        }
    }
}
```

### 使用注意事项

1. **寄存器约定**：
   - 向量寄存器 v0-v63 可以在内联汇编中使用
   - 使用后需要在 clobber 列表中声明
   - 注意保存和恢复调用者保存的寄存器

2. **内存对齐**：
   - 向量加载/存储指令通常要求地址对齐
   - INT8 数据建议 4 字节对齐
   - INT32 数据必须 4 字节对齐

3. **编译器支持**：
   - 需要使用支持 RISC-V 向量扩展的编译器（如 GCC 或 Clang）
   - 编译时需要指定正确的架构参数：`-march=rv32imv`

4. **性能优化**：
   - 尽量使用向量指令处理多个数据
   - 利用 stripmining 机制减少循环开销
   - 注意数据局部性，提高缓存命中率

## 5.4.6 总结

CoralNPU 通过回收 C 扩展的编码空间，为机器学习专用指令预留了充足的编码空间。当前版本主要使用标准的 RVV 指令集，已经能够满足大部分 ML 工作负载的需求。

未来规划的自定义扩展包括：
- **量化指令**：高效的量化和反量化操作
- **MAC 指令**：外积 MAC 引擎，支持高吞吐量的卷积计算
- **数据重排指令**：灵活的数据格式转换

这些自定义指令将进一步提升 CoralNPU 在机器学习推理任务中的性能和能效。
